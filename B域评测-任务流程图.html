L1单点检索 · 一次查询答一个事实（15 题）

policy_qa

×4

问一条现行制度规定（如审批线、报销时限）。

  1. 1检索制度search_policy
  2. 2读原文条款get_policy_docget_policy_excerpt
  3. 3提取事实并作答带 doc_id + clause_no 引用，anomaly_ids 空
判expected_facts 事实命中＋引用正确

single_anomaly_lookup / ground_truth_lookup

×10

题面给线索（记录号 / 发票号），查一条业务事实。

  1. 1按线索取记录get_expense_detaillist_expenses
  2. 2必要时溯发票 / 审批find_invoice_usagelist_approvals
  3. 3读原始事实并作答
判expected_facts

version_check

×1

确认哪一版是现行版（库里有 2025 现行与 2022 废止两版）。

  1. 1列制度看版本list_policy_docs
  2. 2取现行版原文get_policy_doc
  3. 3作答并标明现行勿把 _deprecated 版当依据
判expected_facts
L2制度 × 数据 比对 · 横评核心能力（25 题 · 全是 policy_data_comparison）

policy_data_comparison

×25 · 核心

拿制度规则去核业务数据，以给定线索定位一个异常，返回异常与记录集合。这是横评真正想测的多工具编排。

  1. 1先取制度规则search_policyget_policy_excerpt审批线 / 报销限额 / 时限 / 唯一发票等口径
  2. 2再取业务事实get_expense_detailfind_invoice_usagelist_approvalsget_department_budget
  3. 3规则 × 数据 交叉比对发票是否被多单重复报销、金额是否越过审批档、是否缺审批…
  4. 4判定异常并作答命中则给 anomaly_ids + record_ids + 制度引用
判anomaly_id_set 异常集合与真值完全相等 = 1 分 另算precision / recall
L3全量 / 多跳 / 版本 / 报告 · 深度编排＋抗 context 压力（10 题）

full_year_rule_audit

×5

不给线索，自己全年扫，一个不漏地找出全部某类异常。

  1. 1取规则search_policy
  2. 2分页翻完整年数据list_expensespage 1…N，每页 ≤ 50 → context 压力在此
  3. 3逐条套规则、汇总全部异常
  4. 4输出完整异常集合重点＝召回完整性
判anomaly_id_set（大集合）

two_hop_retrieval

×1

答案要两次链式检索，第一跳的结果是第二跳的输入。

  1. 1第一跳检索search_policyget_expense_detail
  2. 2用其结果构造第二跳get_policy_doc
  3. 3合成两跳结果作答
判expected_facts

version_trap

×1

现行版与废止版并存，引用废止版即判错（红线）。

  1. 1列制度、辨现行 vs 废止list_policy_docs
  2. 2只用现行版get_policy_doc
  3. 3作答命中废止条款须说明其已废止
判expected_facts deprecated_citations 计数

near_clause_disambiguation

×1

检索命中多条相似条款，要选对适用的那条。

  1. 1检索得多条近义条款search_policy
  2. 2读全文消歧get_policy_doc
  3. 3锁定正确条款作答
判expected_facts

audit_report

×2

综合制度＋全量数据，产出结构化审计结论。

  1. 1取规则search_policy
  2. 2翻数据、逐项核查list_expensesget_expense_detail
  3. 3汇总为审计报告作答

TRAP干净但可疑 · 反幻觉：该说"无异常"时别乱报（5 题）

clean_but_suspicious

×5

题面给一条看着可疑、实则合规的记录。正确答案是空异常集 —— 测的是会不会脑补出不存在的异常（假阳性）。

  1. 1取被点名记录的原始事实get_expense_detailfind_invoice_usage
  2. 2取相关制度规则search_policy
  3. 3核对：原始事实＋规则是否共同支持异常不满足 → 判定合规
  4. 4输出空异常集＋说明理由anomaly_ids: []
判no_anomaly 异常集非空 = 假阳性失败
