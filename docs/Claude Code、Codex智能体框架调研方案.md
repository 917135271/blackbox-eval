# Claude Code、Codex、OpenClaude、OpenCode与Oh My Pi智能体框架调研方案

**适用项目：证券费用审计、制度问答与业务数据核查智能体**
**底层模型：DeepSeek V4 Pro**
**实验规模：5个框架、10组配置、15个正式案例，共150次正式任务运行**

## 结论与执行摘要

建议采用“强Coding Agent + 可插拔Skills”作为项目后续技术路线，并同时比较Claude Code Best、Codex、OpenClaude、OpenCode和Oh My Pi五种实现。框架负责通用任务规划、工具调用、Shell、Python、SQLite和工作区管理；制度版本判断、批量费用核查、证据覆盖、误报复核和审计报告等业务能力沉淀为独立Skills。大多数业务规则和流程调整只需修改对应Skill，不需要改动Agent Loop或重写整套提示词。

在此基础上，方案增加“运行治理与上下文管理层”，最终技术路线调整为：

> Coding Agent + 领域Skills + 单题任务状态 + 按需子智能体 + 统一Hooks审计 + 阶段化上下文压缩

六项能力职责不同：Skills保存稳定业务方法；单题任务状态保存当前题目的结构化事实和进度；子智能体负责独立调查、并行执行和复核；Hooks负责强制约束与可审计事件；上下文压缩负责控制模型当前可见内容；Coding Agent负责总体规划、工具编排和最终判断。不能使用其中一项替代其他能力。

本方案的设计依据有三点：

1. 参考五个框架的Skills、Plugin、Extensions和子智能体机制，以及[anthropics/knowledge-work-plugins](https://github.com/anthropics/knowledge-work-plugins)中Finance、Data等插件的组织方法，将业务能力拆分为Skill、References、Scripts、子智能体和校验工具。
2. 根据上一轮55题的实际调用轨迹设计Skills。上一轮主要问题是全量核查不足、`record_ids`遗漏、制度版本判断不完整以及无异常题误报，不是单纯的输出格式问题。
3. 采用共享业务核心和五端适配层。审计规则、SQL/Python方法、路由规则和证据标准只维护一份，再分别包装给五个框架使用，便于后续替换框架或调整功能。

正式实验设置十组：

| 框架 | 基线组 | 领域增强组 |
| --- | --- | --- |
| Claude Code | DeepSeek、完整通用工具、业务MCP、统一结果提交 | 基线能力 + 审计Skills + 受控子智能体 + 证据复核 |
| Codex | DeepSeek、完整通用工具、业务MCP、统一结果提交 | 基线能力 + 等价审计Skills + 受控子智能体 + 证据复核 |
| OpenClaude | DeepSeek、完整通用工具、业务MCP、统一结果提交 | 基线能力 + 等价审计Skills + 受控子智能体 + 证据复核 |
| OpenCode | DeepSeek、完整通用工具、业务MCP、统一结果提交 | 基线能力 + 等价审计Skills + 受控子智能体 + 证据复核 |
| Oh My Pi | DeepSeek、完整通用工具、业务MCP、统一结果提交 | 基线能力 + 等价审计Skills + 受控子智能体 + Hooks + 上下文治理 |
十组分别运行相同的15个正式案例，共150次正式运行。每个案例预先冻结一份100分专属Rubric，按结论、证据、规则、推理和误报控制逐项给分，不再只用整题正确或错误表示能力。基线组用于测量框架本身的任务能力，增强组用于测量Skills带来的实际提升。现有12道独立开发题继续只用于调试工具、Skills、子智能体路由和结果提交机制，不制作逐题Rubric，也不进入正式评分。

子智能体不完全依赖模型自由判断。调用条件、禁止条件、输入范围、输出Schema和Token上限写入Skills，并由统一控制工具执行限制。子智能体不是单纯的上下文缩减手段，只在需要职责隔离、并行调查、不同工具权限或独立复核时使用；一般的信息保存和上下文控制由单题任务状态与压缩机制完成。

## 上一轮实验复盘

上一轮标准55题结果如下：

| 候选 | 正确题数 | 正确率 |
| --- | ---: | ---: |
| trae-agent | 45/55 | 81.8% |
| opencode | 41/55 | 74.5% |
| qwen-code | 33/55 | 60.0% |
| Claude Code | 32/55 | 58.2% |
| Codex | 31/55 | 56.4% |

所有候选格式成功率均为100%，没有超时和Judge错误，说明现有题库、业务MCP、结果采集和判卷链路已经可以支持正式评测。但轨迹显示，Claude Code出现10次`record_id`遗漏和10次无异常题误报，Codex出现8次`record_id`遗漏和13次无异常题误报，主要问题集中在审计过程和证据质量。

| 上一轮轨迹问题 | 本轮对应能力 |
| --- | --- |
| 未先规划完整核查范围，过早形成结论 | `audit-query-planner` |
| 未充分使用SQL/Python执行全量核查 | `batch-expense-analysis` |
| 查询到相关记录但最终遗漏`record_ids` | `evidence-coverage-check` |
| 历史、现行和废止制度判断不完整 | `policy-version-check` |
| 无异常题和陷阱题缺少反向验证 | `false-positive-review` |
| 报告中的事实、制度和结论对应不完整 | `audit-report` |
| 最终结果与中间分析不一致 | `result-validator`和`submit_audit_result` |

因此，本轮的重点是将上一轮暴露的问题转化为可执行Skills，再通过基线组和增强组对比验证这些能力是否真正有效。

## 调研目标

本次调研需要形成三项结论：

1. **路线结论**：领域Skills能否相对框架基线稳定提升正确率，并减少记录遗漏和无异常题误报。
2. **能力结论**：Skills主要改善了哪些题型，子智能体、SQL/Python核查和证据复核是否实际发挥作用。
3. **选型结论**：在相同DeepSeek模型、任务、权限和业务能力下，五个框架中哪个更适合作为后续执行底座。

## 总体技术路线

五个候选均从源码构建并锁定Commit、依赖和构建产物：

- Claude Code：[claude\-code\-best/claude\-code](https://github.com/claude-code-best/claude-code)

- Codex：[openai/codex](https://github.com/openai/codex)

- OpenClaude：[Gitlawb/openclaude](https://github.com/Gitlawb/openclaude)

- OpenCode：[anomalyco/opencode](https://github.com/anomalyco/opencode)

- Oh My Pi：[can1357/oh-my-pi](https://github.com/can1357/oh-my-pi)


- 领域插件参考：[anthropics/knowledge\-work\-plugins](https://github.com/anthropics/knowledge-work-plugins)

整体架构按职责分层：框架负责通用执行，Skills负责审计规则和流程，MCP负责制度及费用数据查询，单题任务状态负责保存当前任务进度，Hooks负责审计事件和强制约束，提交工具负责结果校验。业务规则变化优先修改Skills，数据接口变化调整MCP，避免将所有逻辑集中在框架源码或单一长提示词中。

工程结构：

```Plain Text
domain-agent-eval/
├── shared-audit-core/
│   ├── skills/
│   ├── routing/
│   ├── references/
│   ├── scripts/
│   └── schemas/
├── adapters/
│   ├── claude-code-best/
│   ├── codex/
│   ├── openclaude/
│   ├── opencode/
│   └── oh-my-pi/
├── submit-audit-result/
├── runner/
├── dev-dataset/
├── formal-dataset/
├── configs/
└── output/
```

`shared-audit-core`只保存与框架无关的审计方法、路由规则、SQL/Python脚本、任务状态、事件规范和结果Schema。五个`adapters`只负责转换为对应框架能够加载的Skills、插件、Extensions、Hooks和子智能体配置。

这样设计可以保证五个候选获得相同业务知识，也可以防止同一项审计规则分别维护五份。

## 实验分组与公平性

十组均开放Shell、Python、SQLite、隔离工作区文件读写、现有业务MCP或等价工具桥接，以及统一结果提交工具。

基线组也开放通用工具，是为了测量框架正常工作时的基础能力；增强组只增加领域方法和复核流程。只有保持其他条件一致，增强组与基线组的差值才能用于衡量插件贡献。

公平性要求：

1. 使用同一DeepSeek模型、推理强度和上下文规格。

2. 使用相同任务、制度语料、费用数据库和业务MCP。

3. 使用相同时间、Token和安全轮次上限。

4. 使用相同最终结果Schema和提交工具。

5. 15个正式案例每组每题只运行一次，不择优重跑。

6. 正式运行前冻结源码、配置、Skills和路由文件哈希。

7. 基础设施故障可以记录后重跑，语义错误不得重跑。

Oh My Pi默认具备较丰富的工具、Memory、Advisor、Browser和跨会话能力。正式评测中必须关闭联网搜索、Browser、GitHub、SSH、跨任务`retain/recall/reflect`、历史项目Memory和第二模型Advisor；禁止自动继承隔离工作区以外的Skills、规则和MCP。基线组禁用领域Skills和原生子智能体，增强组只开放统一路由授权的`task`子智能体。其原生Checkpoint、Rewind和上下文压缩可以使用，但必须转换为统一上下文事件，并接受压缩后关键事实保留检查。

## 证券费用审计Skills

证券费用审计Skills是对审计方法、工具使用、子智能体路由、输出结构和停止条件的完整封装。每个Skill至少包含以下内容：

```Plain Text
skills/<skill-name>/
├── SKILL.md                 # 触发条件、执行步骤和完成标准
├── references/              # 制度说明、字段口径和审计方法
├── scripts/                 # 可复用SQL或Python脚本
├── schemas/                 # 中间结果和输出Schema
└── tests/                   # 路由及结果校验测试
```

每个`SKILL.md`统一定义：

### 6\.1 Skills执行顺序

增强组按照以下顺序执行Skills：

1. **审计规划**：首先执行`audit-query-planner`，识别审计对象、时间范围、适用规则、数据字段、证据要求和任务复杂度，并生成子智能体调用计划。

2. **制度与数据核查**：根据审计计划决定是否执行`policy-version-check`和`batch-expense-analysis`。两项任务彼此独立时可以并行执行；简单事实题可以采用精简模式或跳过不相关的Skill。

3. **证据覆盖检查**：制度和数据核查完成后，必须执行`evidence-coverage-check`，检查每个异常项是否具备对应的业务记录和制度依据，并识别已经查询到但未写入结果的记录。

4. **反向复核**：当任务涉及制度例外、金额或日期边界、无异常结论、复杂数据计算或证据冲突时，执行`false-positive-review`，尝试推翻初步结论。符合路由条件时，由该Skill调用独立复核员。

5. **审计报告生成**：证据覆盖检查和反向复核通过后，执行`audit-report`，按照事实、规则、判断、影响和建议组织最终答案。简单题使用精简报告模式，综合审计题使用完整报告模式。

6. **提交前校验**：执行`result-validator`，检查字段类型、异常ID、`record_ids`、制度引用、证据覆盖以及报告正文之间的一致性。首次校验失败后允许主智能体修复一次。

7. **统一结果提交**：校验通过后，调用`submit_audit_result`提交最终结果。提交工具只处理Schema和确定性字段校验，不替智能体修改语义结论。

其中，`audit-query-planner`、`evidence-coverage-check`、`audit-report`和`result-validator`是所有任务的必经步骤；`policy-version-check`、`batch-expense-analysis`和`false-positive-review`根据任务内容和中间结果触发。任何Skill发现证据不足、查询失败或制度适用关系无法确定时，都不得直接进入最终提交，而应返回明确的补充核查项。

### 6\.2 `audit-query-planner`

**目标**

在查询数据前将题目转换为可执行的审计计划，明确规则、对象、范围、字段、证据需求和子智能体调用方式，防止智能体查到少量记录后直接下结论。

**触发条件**

所有正式任务均必须调用。简单事实题可以生成精简计划，但不能完全跳过。

**输入**

- 原始题目；

- 可用制度语料和业务MCP说明；

- 数据库表结构和字段字典；

- 最终结果Schema；

- 当前任务的时间和Token预算。

**执行步骤**

1. 提取审计对象、费用类型、业务期间和组织范围。

2. 识别题目要求判断的规则及边界条件。

3. 明确需要读取的制度、数据表和字段。

4. 将任务拆解为制度核查、数据核查、证据检查和报告生成步骤。

5. 定义每一步的预期输出和停止条件。

6. 计算任务复杂度。

7. 根据路由规则生成子智能体调用计划。

**任务复杂度**

以下每项计1分：

- 涉及制度版本、生效时间或废止关系；

- 涉及两份及以上制度；

- 需要聚合、分组、多表关联或批量记录筛选；

- 同时涉及两条及以上业务规则；

- 存在阈值、日期、审批豁免或例外条件；

- 需要形成综合审计报告。

复杂度为0至1时，原则上由主智能体处理；2至3时最多调用一个专业子智能体；4分及以上时可以调用制度研究员和数据核查员，并在形成初稿后调用独立复核员。

**输出**

生成`audit_plan.json`，包括审计对象、时间范围、制度核查问题、数据核查问题、所需字段、证据要求、任务复杂度、子智能体调用计划和停止条件。

**禁止事项**

- 不得在规划阶段直接生成最终异常结论；

- 不得根据题目表面描述假设数据库中一定存在异常；

- 不得读取Ground Truth或历史答案确定查询范围。

**完成标准**

审计对象、时间范围、核查规则、字段、证据要求和子智能体路由均已明确。

### 6\.3 `policy-version-check`

**目标**

判断业务发生时真正适用的制度和条款，区分现行、历史、废止、替代和过渡制度。

**触发条件**

满足以下任一条件时调用：

- 题目涉及制度生效时间、失效时间或历史版本；

- 检索到多份名称相同或内容相似的制度；

- 业务发生日期可能跨越制度调整时间；

- 制度之间存在冲突、替代或上下位关系；

- 结论依赖例外条款、过渡期或特殊审批条件。

单一现行制度的简单事实定位且已找到明确条款时，可以不调用制度研究员，但仍需执行本Skill的精简检查。

**输入**

- 业务发生日期或核查期间；

- 候选制度ID及元数据；

- 制度正文；

- 需要判断的业务规则；

- 已检索到的相关条款。

**执行步骤**

1. 提取制度发布日期、生效日期、失效日期和替代关系。

2. 建立制度时间轴。

3. 根据业务发生日期筛选适用版本。

4. 检查新旧制度是否存在过渡安排。

5. 检查同一事项是否存在更高层级或更具体的制度。

6. 定位直接支持判断的条款。

7. 记录被排除制度及排除原因。

8. 对无法确定的制度状态标记为`unresolved`，不得自行推断。

**子智能体路由**

以下情况调用制度研究员：

- 候选制度不少于两份；

- 存在历史版本、废止或替代关系；

- 两份制度对同一业务给出不同要求；

- 找到制度正文但无法确定适用期间；

- 最终结论需要引用多个条款。

制度研究员每题最多调用一次。

**输出**

生成`policy_applicability.json`，包括适用制度、被排除制度、制度状态、生效及失效时间、适用条款、版本关系、制度冲突、适用或排除原因，以及尚未解决的问题。

**完成标准**

每个用于最终判断的制度均有明确的适用时间、条款和适用原因；被排除制度也有可检查的排除原因。

### 6\.4 `batch-expense-analysis`

**目标**

使用SQL和Python完成全量业务数据核查，避免只查看少量样例记录或依赖模型逐条阅读。

**触发条件**

- 需要返回多个`record_ids`；

- 需要筛选一定期间内的全部记录；

- 涉及金额求和、分组、频次、比例或阈值；

- 涉及重复报销、拆分报销、跨记录关联；

- 需要关联人员、部门、审批、发票或制度字段；

- 单次MCP查询无法覆盖完整业务范围。

唯一记录ID的单条事实查询可以使用精简模式，不必调用数据核查员。

**输入**

- `audit_plan.json`；

- 数据库表结构和字段字典；

- 只读业务数据库；

- 制度核查结果；

- 题目要求的业务范围。

**执行步骤**

1. 检查表结构、主键、时间字段和金额字段。

2. 设置SQLite只读模式。

3. 先统计核查范围内的总记录数。

4. 编写参数化SQL筛选候选记录。

5. 对聚合、重复、跨期和复杂计算使用Python复核。

6. 对SQL结果与Python结果进行数量及金额核对。

7. 保留所有命中记录，不得只取前若干条。

8. 对`record_ids`去重并检查原始记录是否存在。

9. 保存SQL、参数、结果和统计摘要。

10. 查询失败时标记失败，不得根据部分结果继续作出完整结论。

**子智能体路由**

满足以下任一条件时调用数据核查员：

- 需要聚合或分组；

- 需要两张以上数据表关联；

- 需要处理多条候选记录；

- 需要进行日期区间、金额阈值或重复记录判断；

- SQL执行后仍需Python进行二次计算。

数据核查员每题最多调用一次。

**输出文件**

生成`data_findings.json`，包括核查范围内的记录总数、候选记录数量、命中的`record_ids`、聚合计算结果、已执行检查、查询文件路径、数据异常、查询限制和尚未解决的问题。

SQL、Python脚本和批量查询结果分别保存为独立文件，不直接写入JSON正文。

```Plain Text
work/sql/query.sql
work/analysis/candidate_records.csv
work/analysis/summary.json
work/analysis/data_findings.json
```

**禁止事项**

- 禁止执行`INSERT`、`UPDATE`、`DELETE`或结构修改；

- 禁止只根据查询结果前几行形成结论；

- 禁止在SQL失败后使用估算值替代；

- 禁止生成数据库中不存在的记录ID。

**完成标准**

核查范围、查询逻辑、命中记录和计算过程均可重复执行，所有输出`record_ids`都能追溯到原始数据。

### 6\.5 `evidence-coverage-check`

**目标**

检查每个异常结论是否同时具备事实证据、业务记录和制度依据，并发现已经查询到但最终遗漏的记录。

**触发条件**

所有任务在生成最终报告前必须执行。即使结论为“无异常”，也必须检查查询覆盖范围。

**输入**

- `audit_plan.json`；

- `policy_applicability.json`；

- `data_findings.json`；

- 初步异常结论；

- 待返回的`record_ids`和制度引用。

**执行步骤**

1. 为每个异常项建立证据矩阵。

2. 检查异常ID是否对应明确规则。

3. 检查异常项是否有支持它的业务记录。

4. 检查业务记录是否有适用制度和条款。

5. 检查SQL候选记录是否全部进入最终判断。

6. 检查是否存在无归属的`record_ids`。

7. 检查是否存在未被任何结论使用的制度引用。

8. 检查报告中的金额、数量和日期是否与中间结果一致。

9. 对无异常结论检查核查范围、查询条件和记录数量是否完整。

**输出**

生成`evidence_matrix.json`，包括每个异常项对应的`record_ids`、制度引用、事实支持状态、规则支持状态、证据覆盖状态、证据缺口、无归属记录、未使用候选记录、整体覆盖率和检查结果。

**阻断规则**

以下任一情况不得进入最终提交：

- 异常项没有任何有效业务记录；

- 应引用制度但没有制度依据；

- 存在已确认异常但未写入结果的候选记录；

- 报告中的金额或数量与查询结果不一致；

- 无异常结论没有完整核查范围和查询证明。

**完成标准**

`status=pass`且证据覆盖率达到100%。未通过时返回主智能体补充一次，禁止无限循环。

### 6\.6 `false-positive-review`

**目标**

主动尝试推翻初步结论，检查异常是否由制度例外、边界条件、数据重复或查询口径错误造成，同时检查无异常结论是否存在漏查。

**触发条件**

满足以下任一条件时调用：

- 初步结论包含异常，且规则存在例外或审批豁免；

- 金额、日期或次数接近制度阈值；

- 涉及历史制度、过渡期或跨期业务；

- 制度研究结果与数据核查结果不完全一致；

- `evidence-coverage-check`发现缺口；

- 综合审计题；

- 初步结论为无异常，但核查范围较大或规则较多。

**执行步骤**

1. 为每项异常建立“该记录其实合规”的反向假设。

2. 检查金额边界是否包含等于条件。

3. 检查日期边界、时区和制度生效日。

4. 检查审批豁免、特殊事项和例外条款。

5. 检查重复数据、冲销记录、撤销记录和空值。

6. 检查费用分类或人员身份是否被错误匹配。

7. 检查跨记录聚合是否重复计算。

8. 对无异常结论重新检查查询范围和遗漏条件。

9. 输出通过、驳回或补充查询意见。

**子智能体路由**

达到触发条件时调用独立复核员。复核员只接收初步结论、证据矩阵、必要制度条款和中间结果路径，不接收主智能体完整轨迹。

独立复核员不得继续派生子智能体，每题最多调用一次。

**输出**

生成`independent_review.json`，包括复核结论、被质疑的异常项、支持异常的证据、反对异常的证据、发现的例外或边界条件、需要修改的内容、需要补充的查询和复核理由。

**处理规则**

- `pass`：可以继续生成报告；

- `reject`：必须删除或修正被驳回的异常；

- `needs_more_evidence`：允许补充一次指定查询；

- 补充查询后不得再次启动独立复核员。

**完成标准**

所有关键异常和无异常结论都经过反向验证，未解决冲突不得提交。

### 6\.7 `audit-report`

**目标**

将通过证据检查和独立复核的结果组织成结构清晰、事实与判断分离的审计结论。

**触发条件**

所有任务均执行。简单题使用精简报告，综合题使用完整报告。

**输入**

- 审计计划；

- 适用制度；

- 数据核查结果；

- 证据矩阵；

- 独立复核意见；

- 最终异常项和`record_ids`。

**报告结构**

异常题按照以下结构：

1. 核查范围；

2. 已确认事实；

3. 适用制度和条款；

4. 异常判断；

5. 涉及记录和影响范围；

6. 处理或整改建议。

无异常题按照以下结构：

1. 核查对象和期间；

2. 执行的查询和核查方法；

3. 适用制度；

4. 未发现异常的结论；

5. 尚未覆盖的数据限制，如存在。

**输出要求**

- 事实、规则和判断分段表达；

- 每个异常结论必须能对应`record_ids`；

- 不得在报告阶段新增未经核查的异常；

- 不得生成中间结果中不存在的金额、日期或ID；

- 不得使用“可能存在大量问题”等无证据扩大表述；

- 简单题保持简洁，综合题完整说明过程和依据。

**输出**

生成`report_draft.json`，包括核查范围、事实发现、适用制度、异常判断、影响范围、处理建议、`anomaly_ids`、`record_ids`、报告正文和制度引用。

简单题可以只保留必要内容，综合审计题应完整填写事实、规则、判断、影响和建议。

**完成标准**

报告内容与证据矩阵一致，不存在无记录、无制度或无事实支持的结论。

### 6\.8 `result-validator`

**目标**

在调用`submit_audit_result`前，对待提交结果执行最后一次确定性检查。

**触发条件**

所有任务必须执行，且每题只能完成一次正式校验和一次修复后复验。

**输入**

- `report_draft.json`；

- `evidence_matrix.json`；

- 可用异常ID、记录ID和制度ID清单；

- 最终提交Schema。

**检查内容**

1. 必填字段是否完整；

2. 字段类型是否正确；

3. 数组是否存在空值和重复项；

4. 异常ID、记录ID和制度ID是否真实存在；

5. `anomaly_ids`、`record_ids`和报告正文是否相互矛盾；

6. 报告声称“无异常”时，异常数组是否为空；

7. 报告声称存在异常时，是否有异常ID和证据；

8. 每个异常是否通过证据覆盖检查；

9. 制度引用是否来自适用制度；

10. 最终记录集合是否遗漏已确认异常记录。

**输出**

生成`validation_report.json`，包括校验状态、错误列表、警告列表、字段检查结果、ID存在性检查结果、证据覆盖检查结果、正文一致性检查结果、是否允许修复，以及需要修复的字段。

校验通过后，另行生成符合`submit_audit_result`要求的最终提交JSON。

**修复机制**

- 第一次校验失败后，向主智能体返回结构化错误；

- 主智能体只允许修改错误涉及的字段；

- 修复后再校验一次；

- 第二次仍失败则按格式或证据校验失败提交，不继续循环。

**禁止事项**

`result-validator`不能自行增加异常、删除异常或推断缺失记录，只能指出错误并要求主智能体修复。

**完成标准**

`validation_report.status=pass`后，才能调用`submit_audit_result`。



## 子智能体受控路由设计

### 7\.1 路由方式

子智能体不能只由模型自由判断是否调用。具体采用三层控制：

1. `audit-query-planner`根据题目和初步检索结果输出结构化`subagent_plan`。

2. Skills中固化每个子智能体的触发条件和禁止条件。

3. `authorize_audit_subagent`检查调用是否合法，并执行角色、次数、上下文、产物和Token限制。

路由结果示例：

```JSON
{
  "policy_researcher": {
    "required": true,
    "reason_code": "MULTI_POLICY_VERSION_CHECK"
  },
  "data_analyst": {
    "required": true,
    "reason_code": "BATCH_AGGREGATION_REQUIRED"
  },
  "independent_reviewer": {
    "required": false,
    "reason_code": "WAIT_FOR_DRAFT"
  }
}
```

Reason Code采用固定枚举，便于统计子智能体为什么被调用，而不是只保留自然语言理由。

### 7\.2 调用条件

运行时不能读取正式题型标签或Ground Truth决定路由，只能依据题目内容、查询计划、中间结果和待提交答案触发。

### 7\.3 子智能体职责与产出

| 子智能体 | 触发时间 | 主要职责 | 必须形成的产物 |
| --- | --- | --- | --- |
| 制度研究员 | 存在历史制度、多版本冲突、过渡期、例外条款或多条款组合 | 判断制度效力、适用期间、冲突关系和例外条件 | `work/subagents/policy_researcher/policy_applicability.json`及必要制度摘录 |
| 数据核查员 | 需要全量扫描、跨记录聚合、多表关联、日期金额边界或Python复算 | 执行完整SQL/Python核查并核对统计口径 | SQL脚本、完整查询结果、`data_findings.json`及复算文件 |
| 独立复核员 | 初步结论和证据矩阵形成后，存在无异常结论、边界、例外或综合审计 | 尝试推翻初步结论，识别误报、遗漏和证据缺口 | `independent_review.json`及补证清单 |

子智能体的必要性不是“上下文太长”，而是：

- 制度核查和数据核查可以并行执行；
- 不同角色可以使用不同工具和权限；
- 独立复核员不接收主智能体完整推理，降低结论锚定；
- 每个角色具有独立产物和责任边界，便于失败归因；
- 每项调查可以单独限制Token、轮次和调用次数。

单条制度查询、单条SQL、简单事实核对以及仅因上下文较长的任务，不调用子智能体，由主智能体、Skills和单题任务状态完成。由于主智能体和子智能体均使用DeepSeek，所谓“独立复核”是角色、上下文和工具权限隔离，不代表模型层面的完全独立。

### 7\.4 子智能体完成协议

子智能体采用严格的两阶段控制协议：

1. 主智能体调用`authorize_audit_subagent`，提交角色、复杂度、Reason Code、最小上下文、可读产物路径和Token预算。
2. 控制工具返回唯一`invocation_id`和派发契约，框架随后调用原生子智能体。
3. 子智能体将完整分析和产物写入`work/subagents/<role>/`。
4. 主智能体调用`complete_audit_subagent`，提交`invocation_id`、摘要文件和产物路径。
5. 控制工具校验摘要Schema、产物存在性、文件哈希、角色状态和Token使用，并登记完成事件。
6. 主智能体只接收验证通过的结构化摘要；未完成登记的子智能体结果不得进入最终证据矩阵。

当前`run_audit_subagent`只完成授权并返回派发契约，名称容易被理解为已经执行子智能体。实施时将其重命名为`authorize_audit_subagent`，并新增`complete_audit_subagent`。

返回主智能体的摘要只包括：

- `decision`
- `key_findings`
- `record_ids`
- `citations`
- `unresolved_items`
- `artifact_paths`

### 7\.5 上下文控制

主智能体不得把完整历史对话直接发送给子智能体。子智能体只接收：

- 当前问题；

- 与角色相关的任务说明；

- 必要的制度ID、字段、记录范围或中间文件路径；

- 固定输出Schema；

- 本角色的Token和工具权限。

子智能体的完整分析保存在`work/subagents/`中，返回主智能体的内容限制为结构化摘要：

```JSON
{
  "decision": "pass|reject|needs_more_evidence",
  "key_findings": [],
  "record_ids": [],
  "citations": [],
  "unresolved_items": [],
  "artifact_paths": []
}
```

返回主上下文的摘要不超过800至1200 Token。SQL结果、大批记录和长制度原文只保存为文件，主智能体按需读取，不直接注入上下文。

### 7\.6 单题任务状态

本方案不引入跨题长期记忆，也不需要为15分钟单题任务建设向量数据库。Memory分为三类：

- 领域记忆：Skills、制度说明和SQL方法，正式运行前冻结；
- 单题任务状态：当前题目的计划、发现、证据、预算和未决事项，允许使用；
- 跨题记忆：其他题目的答案、轨迹和经验，正式评测中禁止。

单题任务状态采用文件化结构：

- `work/task_state.json`：当前阶段、范围、预算和完成状态；
- `work/evidence_index.json`：记录、制度和异常项之间的索引；
- `work/decision_log.jsonl`：关键判断、依据和变更原因；
- `work/context_checkpoint.json`：压缩后必须恢复的最小上下文；
- `work/artifact_index.json`：全部中间产物、哈希和所有者。

单题任务状态保存“已经确认了什么、还缺什么、产物在哪里”，不保存其他题目的答案，也不直接替代子智能体执行调查。

### 7\.7 上下文Checkpoint与压缩

上下文控制同时使用阶段触发和容量触发。

阶段触发：

1. 审计规划完成后生成第一次Checkpoint；
2. 制度与数据调查完成后生成第二次Checkpoint；
3. 独立复核前生成第三次Checkpoint；
4. 最终校验前生成提交Checkpoint。

容量触发：

- 活跃上下文估算达到60%时写Checkpoint；
- 达到75%时执行压缩；
- 达到85%时禁止宽泛检索，只允许定向补证、校验和提交。

如果框架不能提供可靠的上下文Token占用，则使用代理估算值，包括累计输入Token、当前消息长度、工具返回长度和已加载文件大小。具体阈值在开发题中校准，正式运行前冻结，不能在GATE4期间动态修改。

压缩后必须保留题目、限制条件、审计计划、适用制度、候选及最终记录ID、证据状态、未解决问题、产物索引、剩余预算和提交状态。SQL原始行、制度长原文、已完成工具对话和子智能体完整轨迹从活跃上下文移除，但对应文件和日志不得删除。

### 7\.8 Hooks与统一审计事件

Hooks采用两层设计：

1. 外层统一Runner作为权威审计层，记录所有框架的进程、模型、工具、文件、Token、耗时、退出状态和提交事件。
2. 框架原生Hooks或事件机制补充记录Skill、原生子智能体和上下文事件，并转换为统一事件格式。

至少记录以下事件：

- `task_started`
- `skill_selected`
- `tool_started`、`tool_completed`、`tool_failed`
- `subagent_requested`、`subagent_authorized`、`subagent_rejected`、`subagent_completed`
- `artifact_written`
- `context_checkpoint`、`context_compacted`
- `validation_started`、`validation_completed`
- `submission_attempted`、`submission_accepted`、`submission_rejected`
- `task_completed`、`task_timeout`

每条事件包含任务ID、框架、实验组、角色、时间、递增序号、原因码、输入输出摘要、产物路径、文件哈希、Token、耗时和错误码。日志不得写入密钥；长输出只保留摘要、路径和哈希。

统一产物包括：

- `traces/events.jsonl`
- `traces/tool_calls.jsonl`
- `traces/subagents.jsonl`
- `traces/context_events.jsonl`
- `traces/artifact_manifest.json`
- `run_manifest.json`

权限检查、子智能体授权与完成登记、工作区边界和结果提交属于强制执行点，失败时阻断继续执行。普通统计日志写入失败时任务可以继续，但本次运行标记为基础设施异常；正式结果不得在审计事件不完整时被解释为完整成功。

## 标准审计执行流程

增强组按照以下流程运行：

1. **任务解析**：识别审计对象、时间范围、制度范围和预期输出。

2. **审计规划**：`audit-query-planner`生成核查计划和子智能体路由计划。

3. **制度核查**：必要时调用制度研究员，确认适用制度和例外条款。

4. **数据核查**：必要时调用数据核查员，通过SQL/Python完成全量分析。

5. **初步结论**：主智能体形成异常项、记录和制度依据。

6. **证据覆盖检查**：检查异常、`record_ids`和制度引用是否一一对应。

7. **反向验证**：满足路由条件时调用独立复核员。

8. **报告生成**：形成事实、规则、判断、影响和建议。

9. **结果校验**：`result-validator`检查待提交结果。

10. **统一提交**：通过`submit_audit_result`提交，失败时只允许修复一次。

这个流程将调查和提交分开，可以避免模型在查到第一条可疑记录后立即结束任务。

## 统一结果提交工具

五个候选使用相同Schema：

```JSON
{
  "anomaly_ids": ["A001"],
  "record_ids": ["R001", "R002"],
  "answer": "审计结论及依据",
  "citations": [
    {"doc_id": "POLICY_001", "clause_no": "第三条"}
  ]
}
```

`submit_audit_result`负责：

1. JSON Schema校验；

2. 必填字段和字段类型校验；

3. 数组去重和空值检查；

4. 记录ID、异常ID和制度ID存在性检查；

5. 证据覆盖提示；

6. 返回结构化错误；

7. 允许智能体自动修复并再次提交一次。

该工具不能自行增加异常、补充记录或改变语义结论。这样可以把答案内容错误和结果格式错误分别统计。

## 指标与量化评分方法

最终总分为100分：

$Total = 0.55Q + 0.20E + 0.15S + 0.10F$

各子分先计算为0至100分，再乘对应权重。

### 12\.1 任务正确性Q，占55%

$ Q = \frac{\sum_{i=1}^{15} CaseScore_i}{15} $

每题`CaseScore`为该题专属Rubric各评分点得分之和，范围为0至100分。每个评分点必须记录得分、依据和对应的答案或轨迹证据。单题达到70分且没有触发关键错误封顶时记为通过，但最终比较使用连续分数，不把69分与0分视为相同错误。

关键错误封顶规则也按题冻结。例如：陷阱题误报实质异常时该题最高30分；全年扫描遗漏整类异常时该题最高60分；使用废止制度形成当前结论时该题最高40分；伪造数据库不存在的`record_id`时该题最高20分。

### 12\.2 证据质量E，占20%

$E = 0.40R + 0.30C + 0.30N$

其中：

- $R$：Record ID正确率；

- $C$：制度引用正确率；

- $N$：无异常及陷阱题正确率。

即：

- $ R = \frac{record\_ids完全正确的题数}{要求返回record\_ids的题数} \times 100 $

- $C = \frac{制度引用正确的题数}{要求引用制度的题数} \times 100$

- $N = \frac{正确判断的无异常及陷阱题数}{无异常及陷阱题总数} \times 100$

### 12\.3 稳定性S，占15%

$S = \frac{有效完成任务数}{15} \times 100$

一项任务同时满足以下条件，才算有效完成：

- 在规定时间内完成；

- 成功提交最终结果；

- 最终结果通过Schema校验；

- 没有因进程、MCP或工具错误导致任务中断。

### 12\.4 执行效率F，占10%

$ F = 0.50T + 0.50K$

其中：

- $T$：时间得分；

- $K$：Token得分。

## 判卷机制

判卷采用三层机制：

1. 规则程序计算Schema、异常ID、`record_ids`、引用集合和量化指标。

2. DeepSeek LLM Judge逐项读取本题冻结Rubric，输出每个评分点的得分、理由、答案证据和轨迹证据。

3. 人工复核所有规则与Judge不一致题、全部陷阱题错误、全部综合审计题，以及20%的分层随机样本。

每题保留0至100分、是否通过、关键错误和逐评分点明细。规则和Judge存在冲突时，以人工依据冻结Rubric作出的结论为准，不允许判卷后修改Rubric迁就某个候选答案。

硬性成功标准：

- 格式成功率100%；

- 每组超时不超过1题；

- 陷阱题和无异常题误报不超过1题；

- 增强组相对同框架基线的任务正确性Q至少提升8分，并且至少四类案例取得正向提升；

- `record_id`遗漏和无异常误报较上一轮下降至少50%；

- 子智能体原始轨迹泄漏率为0。

新增诊断指标，不计入总分：

- 子智能体路由合规率；
- 简单题无效子智能体调用率；
- 子智能体产物完整率；
- 子智能体完成登记率；
- 独立复核纠错率；
- 审计事件完整率；
- Checkpoint生成合规率；
- 上下文压缩次数；
- 压缩后关键事实保留率；
- 子智能体带来的正确率、耗时和Token变化。

## 最终选型规则

首先检查硬性标准。未达到格式、超时、误报和轨迹隔离要求的候选，不进入最终推荐。

通过硬性标准后，按照以下公式计算：

$Total = 0.55Q + 0.20E + 0.15S + 0.10F$

最终同时比较：

- 五个增强组的总分和15题Rubric平均分；

- 每个框架增强组相对自身基线组的Q值、总分和各案例类型提升；

- 五个增强组在制度版本、全年扫描、陷阱题和综合报告上的最低表现；

- 子智能体是否真正修正错误，而不是只增加调用和上下文。

排名第一与第二的增强组总分相差3分及以上时，推荐总分较高者；差距小于3分时，优先选择证据完整性得分更高、无异常误报更少、低分案例更少的候选。

- 如果至少三个框架的增强组相对各自基线Q值提升8分以上，说明“强Coding Agent + 领域Skills + 受控子智能体”路线具备跨框架可迁移性。

- 如果只有一个达到标准，则说明该框架更适合承载当前领域工作流。

- 如果只有个别框架达到标准，说明领域能力有效但依赖具体框架的Skill加载和工具编排方式。

- 如果五个框架均未达到，应先调整Skills、路由、数据工具和复核机制，再决定框架，而不能只依据增强组总分定型。
