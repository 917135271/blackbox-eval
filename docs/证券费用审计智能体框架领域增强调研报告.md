## 一、调研背景、问题来源与本轮目标

上一轮标准55题黑盒评测中，所有候选格式成功率均为100%，没有超时和Judge错误，说明题库、MCP、结果采集和判卷链路已经具备基础可用性。但该结果不能直接解释为Claude Code和Codex框架能力较弱：当时统一使用DeepSeek后端和MCP-only工具模式，最多执行12步，文件、Shell、子智能体和工作区能力受到限制，并要求智能体直接生成严格JSON。Claude Code和Codex均未按完整Coding Agent方式运行，因此上一轮结果应定位为受限条件下的兼容性基线，而不是框架完整能力排名。

上一轮更重要的产出是完整调用轨迹。轨迹表明，主要问题不是输出格式，而是审计过程和证据质量不完整。其中Claude Code出现10次`record_id`遗漏和10次无异常题误报，Codex出现8次`record_id`遗漏和13次无异常题误报。本轮据此将轨迹问题逐项转化为可执行、可检查的领域能力：

| 上一轮轨迹问题 | 对审计结果的影响 | 本轮对应能力 |
| --- | --- | --- |
| 未先规划完整核查范围，查到少量记录后过早下结论 | 漏查对象、期间、规则或数据字段 | `audit-query-planner` |
| 未充分使用SQL或Python执行全量业务核查 | 以抽样结果代替全量审计 | `batch-expense-analysis` |
| 已查询到相关记录，但最终遗漏`record_ids` | 结论无法追溯到完整业务记录 | `evidence-coverage-check` |
| 历史、现行和废止制度的适用关系判断不完整 | 引用错误版本或忽略过渡期与例外 | `policy-version-check` |
| 无异常题和陷阱题缺少反向验证 | 将相似记录误判为异常 | `false-positive-review` |
| 报告中的事实、制度、判断和影响没有完整对应 | 报告结论有方向但证据链不闭合 | `audit-report` |
| 最终提交与中间分析不一致 | 异常ID、记录ID、引用或正文在提交时丢失 | `result-validator`与`submit_audit_result` |

因此，本轮不再单独比较Agent Loop，而是比较由Coding Agent、领域Skills、任务状态、复核机制、上下文管理和统一治理共同组成的完整方案，重点回答三个问题：

1. 领域增强能否改善任务正确性、证据完整性和误报控制。
2. B组完整Skills和C组脚本下沉分别改善了哪些题型、审计环节和框架，哪些场景反而产生负优化。
3. 在相同模型、任务、权限和评分标准下，哪个“框架 + 增强包装”更适合作为后续执行底座。

## 二、技术路线及其必要性

本轮采用“强Coding Agent + 可插拔业务能力”作为核心技术路线。证券费用审计不是单轮问答，而是由制度检索与版本判断、SQL聚合、Python复算、证据整理、反向复核和结构化提交共同组成的执行型任务。成熟Coding Agent可以直接承担规划、工具调用、脚本执行和工作区管理；领域Skills则把审计口径和标准流程固化为可检查、可修改、可复用的业务能力。这样既保留成熟框架的通用执行能力，也避免从零开发Agent Loop。

在此基础上，本轮增加运行治理与上下文管理，形成完整架构：

> Coding Agent + 领域Skills + 单题任务状态 + 按需子智能体 + 统一Hooks审计 + 阶段化上下文管理 + 确定性工作流脚本 + 统一结果提交

各层解决的问题不同，不能用一个长提示词或单一Memory机制替代：

| 能力层 | 主要职责 | 边界 |
| --- | --- | --- |
| Coding Agent | 总体规划、工具编排、文件处理和最终业务决策 | 不固化具体证券审计口径 |
| 领域Skills | 保存稳定的审计方法、检查顺序、规则和产物要求 | 不保存其他题答案，不替代实际数据查询 |
| 单题任务状态 | 保存当前题的计划、证据、未决事项、预算和提交状态 | 不形成跨题记忆 |
| 按需子智能体 | 承担复杂制度研究、批量数据核查或独立复核 | 简单检索和单条SQL不调用，不只为缩短上下文而调用 |
| Hooks与控制MCP | 记录关键事件，执行权限、路由、产物和提交约束 | 不替代模型的业务判断 |
| 上下文管理 | 通过Checkpoint和结构化摘要控制模型当前可见内容并支持恢复 | 不删除原始审计证据；原生压缩能力仍取决于框架 |
| 确定性工作流脚本 | 固化路由、状态快照、集合校验、引用规范化和提交动作 | 不判断制度适用性、异常事实或证据归属 |
| 统一提交工具 | 校验Schema、异常ID、记录ID、引用和证据完整性 | 不根据格式校验结果推断语义正确性 |

本轮设置A、B、C三种配置。A组保留Coding Agent、基础工具和业务MCP，用于测量框架基础能力；B组增加7个完整Skills、Task Memory、按需子智能体、Hooks和结果校验，由模型编排完整审计流程；C组保留5个薄Skills，将路由、状态、Checkpoint、证据集合校验和提交下沉为确定性脚本，模型继续负责制度适用、数据边界、异常认定和报告语义。B、C是两种并列增强包装，不预设哪一种天然更优，最终根据各框架实测结果选择。

本轮设计依据主要有三点：

1. **参考成熟插件与Skills的组织方法。** 参考六个框架可用的Skills、Plugin、Extensions和子智能体机制，以及Anthropic `knowledge-work-plugins`中Finance、Data等插件的职责拆分方式，将能力组织为Skills、参考资料、脚本、子智能体和校验工具，而不是不加选择地安装整套插件。
2. **根据上一轮真实轨迹设计。** 每个核心Skill都对应已经出现的全量核查不足、制度版本误判、记录遗漏、无异常误报或提交不一致问题，避免将领域增强写成泛化提示词。
3. **采用共享业务核心和六端适配层。** 审计规则、SQL/Python方法、路由规则、证据标准和结果Schema只维护一份，六个框架仅维护加载与调用适配。制度口径或核查流程变化时，优先修改对应Skill及其脚本，不需要改写Agent Loop。

A/B/C三组使用相同DeepSeek模型、业务数据、MCP、Shell/Python/SQLite权限、隔离工作区、正式案例、逐题Rubric和评分公式，仅改变领域能力的装配与职责分配。所有Skills和脚本均根据上一轮轨迹与独立开发题形成，不读取正式题答案；正式任务、Rubric和Ground Truth在GATE3配置冻结前均不向候选开放。由此可以分别观察框架基础能力、完整Skills贡献和脚本下沉贡献，并将最终选型落实为“框架 + 增强包装”。

## 四、领域增强能力设计

### 4.1 七个核心Skills

| Skill | 触发场景 | 执行动作 | JSON产物包含 |
| --- | --- | --- | --- |
| `audit-query-planner` | 所有增强任务首先执行 | 拆解对象、期间、规则、范围、字段和证据需求 | 审计范围、规则清单、查询计划、证据需求、未决项 |
| `policy-version-check` | 历史制度、版本冲突、过渡期或例外条款 | 比较生效、废止、替代关系，确定适用条款 | 适用制度、排除制度、适用期间、引用、冲突与结论 |
| `batch-expense-analysis` | 全年扫描、跨记录聚合、多表关联或边界复算 | 使用SQL/Python全量查询并保存完整结果 | 查询口径、SQL或脚本路径、候选记录、异常记录、统计与复算结果 |
| `evidence-coverage-check` | 已形成候选异常后 | 核对每项发现的记录ID、制度依据和解释是否齐全 | 发现清单、证据覆盖状态、缺失记录、缺失引用、补证动作 |
| `false-positive-review` | 无异常、陷阱、边界、例外或高影响结论 | 反向验证是否存在误报、范围扩大或规则混用 | 保留项、驳回项、边界说明、误报风险和复核结论 |
| `audit-report` | 证据矩阵完成后 | 将事实、规则、判断、影响和建议逐项对应 | 审计结论、异常明细、制度依据、影响、建议和限制 |
| `result-validator` | 正式提交前 | 检查Schema、异常ID、记录ID、引用和正文一致性 | 校验状态、错误、警告、修复项和可提交状态 |

Skills按任务复杂度执行。简单制度题可以在规划后直接进行版本核查和提交；数据题必须执行批量核查与证据覆盖检查；陷阱题必须执行误报复核；综合报告题必须形成证据矩阵、报告和提交前校验。这样既避免所有题机械调用全部Skills，也保留关键审计控制点。

### 4.2 B组与C组的增强方式差异

B组和C组都属于领域增强组，使用相同模型、题目、业务数据、MCP、工具权限和提交Schema；差异不在业务规则，而在确定性流程由模型还是脚本承担。

| 能力环节 | B组：原Skills增强 | C组：脚本下沉增强 |
| --- | --- | --- |
| 业务判断 | 模型负责制度适用、数据分析、异常认定和报告语义 | 与B组相同，仍由模型负责，脚本不推断业务事实 |
| Skills包装 | 使用7个完整Skills，由模型选择、编排并执行各阶段 | 保留规划、制度版本、批量分析、误报复核、审计报告5个薄Skills |
| 流程与状态 | 模型按Skills维护Task Memory、Checkpoint和阶段衔接 | 脚本固定路由并生成初始/最终状态快照和阶段Checkpoint |
| 证据处理 | 模型通过`evidence-coverage-check`整理并检查证据 | 模型填写逐异常证据或无异常核查证明，脚本检查异常、记录和引用集合一致性 |
| 结果校验与提交 | 模型执行`result-validator`并调用统一提交工具 | 脚本执行结构校验、引用规范化和无参数文件提交 |
| 子智能体 | 按路由规则调用，用于制度研究、数据核查和独立复核 | 仅在需要独立语义调查时可选调用，不承担脚本已经固化的控制动作 |
| 实验目的 | 测量完整领域增强包相对A组的净变化 | 验证将确定性职责从模型下沉到脚本后是否改善正确性、稳定性和耗时 |

C组使用`scripted-enhancement-v1.2`。无异常题必须由模型填写实际核查范围和查询条件；有异常题必须由模型完成每项异常与实际记录、制度条款和事实的对应。脚本只在模型提供语义证据后执行归一化与一致性检查，不能根据一句结论自动补造完整证据。因此，C-B差值反映两套增强包装的整体差异，不能解释为单一脚本因素的因果贡献。

### 4.3 子智能体协议

| 子智能体 | 何时调用 | 明确产出 | 何时返回主智能体 |
| --- | --- | --- | --- |
| 制度研究员 | 多版本冲突、过渡期、例外条款或多制度组合 | 适用与排除制度、期间、条款引用、冲突和未决问题 | 制度范围已收敛，引用可定位，未决问题已列明时 |
| 数据核查员 | 全量扫描、跨记录聚合、多表关联、金额或日期边界复算 | SQL/Python脚本、完整结果集、统计口径、候选与异常记录 | 查询范围完整、结果已落盘并生成摘要时 |
| 独立复核员 | 初步结论形成后，尤其是无异常、边界和综合审计题 | 对每项发现的保留、驳回或补证意见，以及遗漏和误报清单 | 所有初步发现均有复核状态，重大异议已明确时 |

调用采用两阶段闭环：主智能体先通过`authorize_audit_subagent`提交角色、复杂度、原因码、输入范围和Token预算；授权后由框架执行原生子智能体；完成时调用`complete_audit_subagent`校验结果字段、产物路径、文件哈希、Token和状态；主智能体只接收结构化摘要，包括`decision`、`key_findings`、`record_ids`、`citations`、`unresolved_items`和`artifact_paths`。完整分析留在`work/subagents/<role>/`，需要时再按路径读取。

子智能体不是为了解决一般性的上下文过长。单条制度检索、单条SQL或简单复算由主智能体和Skill完成。只有任务可并行、需要独立权限或需要反向复核时才调用。由于主智能体与子智能体都使用DeepSeek，本轮的“独立”是角色和上下文隔离，不是模型层面的完全独立。

### 4.4 Task Memory与上下文管理

Memory分为三类：领域记忆保存冻结的Skills、制度方法、SQL模板和Schema；任务工作记忆保存当前题的计划、证据和未决项；跨任务记忆包含其他题答案和历史经验，在正式评测中禁止。

本轮不引入向量数据库，每题使用文件化Task Memory：

| 文件 | 保存内容 |
| --- | --- |
| `work/task_state.json` | 当前阶段、范围、预算、待办、未决项和提交状态 |
| `work/evidence_index.json` | 发现、`record_ids`、制度引用、证据状态和产物位置 |
| `work/decision_log.jsonl` | 关键判断、依据、变更原因和责任角色 |
| `work/context_checkpoint.json` | 恢复任务所需的压缩状态 |
| `work/artifact_index.json` | SQL、脚本、结果、报告及哈希索引 |

上下文管理采用阶段触发和容量触发。完成审计规划、制度与数据调查、独立复核时分别写Checkpoint；估算上下文达到60%时强制写Checkpoint，达到75%时请求压缩，达到85%时停止宽泛检索，只允许定向补证、校验和提交。压缩后必须保留题目限制、计划、适用制度、记录ID、证据状态、未决问题、产物索引、剩余预算和提交状态；长原文、原始SQL行和已完成工具对话退出活跃上下文，但原文件不删除。

需要明确实现边界：统一治理层已经验证Checkpoint路径、字段、事件和任务恢复，但第三方框架内部已经持有的对话历史，只能依赖各框架原生压缩能力处理。本轮没有采集到统一的`context_compacted`事件，因此只能证明“状态可恢复”，不能证明六个框架都在75%阈值执行了原生压缩。后续若正式比较上下文治理，应统一记录估算方法、压缩前后Token和关键字段保留结果。

### 4.5 Hooks与可审计日志

Hooks采用两层实现。外层Runner统一记录进程启动、工具、文件、耗时、超时和提交结果；框架原生Hook及控制MCP记录Skill、子智能体、Checkpoint、校验和提交等业务事件。

关键事件包括`task_started`、`skill_selected`、`tool_started/completed/failed`、`subagent_requested/authorized/rejected/completed`、`artifact_written`、`context_checkpoint/context_compacted`、`validation_started/completed`、`submission_attempted/accepted/rejected`和`task_completed/timeout`。

每条事件包含任务ID、框架、实验组、角色、时间、序号、原因码、输入输出摘要、产物路径、Token、耗时和错误码。长输出只记录摘要、路径和哈希，密钥特征自动脱敏。最终形成：

- `traces/events.jsonl`
- `traces/tool_calls.jsonl`
- `traces/subagents.jsonl`
- `traces/context_events.jsonl`
- `traces/artifact_manifest.json`
- `run_manifest.json`

权限、子智能体授权、结果校验和正式提交属于强制Hook，失败时阻断流程；普通统计日志失败不改变业务结论，但正式运行标记为基础设施故障。

## 五、数据集、Rubric与实验流程

### 5.1 正式题设计

修复后的正式集版本为`securities-expense-audit-formal-15-v9`，共15题，覆盖五类任务，每类3题：制度与版本判断、单案数据核查、全年批量审计、无异常及陷阱题、检索与综合报告。开发题仅用于工具和流程调试，不参与正式评分。

针对历史结果中4道全组满分题，本版不再只修改措辞，而是更换业务对象和Ground Truth：`PV-CASE-001`替换`L1-001`，核查9990元审批与住宿双边界；`RA-CASE-001`替换`L2-003`，使用新的重复发票记录组；`RA-CASE-002`替换`L2-008`，使用三笔四天合计10200元的拆分组合；`RR-CASE-001`替换`L3-010`，使用四笔七天合计9764.73元的版本误报边界。四题均由正式SQLite数据库复算记录集合、日期、金额、发票和审批事实。

每题均配置独立的case-by-case Rubric，而不是将一套通用标准机械套用到所有题目。Rubric版本为`atomic-binary-checklist-v7`，共251个原子检查项，单题8至27项。设计过程分为五步：

1. **确定题目边界。** 根据题目明确核查对象、时间范围、业务规则和期望输出，避免把题目未要求的内容纳入评分。
2. **锁定事实依据。** 将制度文件及条款、业务数据库记录、异常类型和Ground Truth映射为可核验依据；这些内容只供判卷使用，不向候选开放。
3. **拆解原子检查项。** 将一个完整答案拆成结论、证据、制度、推理、误报控制和格式六类检查点，每项只判断一个事实或约束，避免“结论基本正确”这类模糊评分。
4. **写明通过条件。** 每项固定`expected`、`evidence_sources`、`pass_condition`和`fail_condition`。只有完全满足通过条件才记1分，否则记0分，不给小数分，也不因文风、篇幅或异常ID命名差异扣分。
5. **选择判定方式。** 集合、数量、精确率、召回率和Schema等客观项由规则最终判断，不进入Judge申诉；制度适用性和业务推理交给Judge。固定抽样20%的语义项进行第二次判卷，分歧时执行第三次裁决并报告一致率。

251个检查项的组成如下：

| 评分维度 | 检查项数 | 主要核查内容 |
| --- | ---: | --- |
| 结论`conclusion` | 16 | 是否识别异常、异常类型和异常数量是否正确 |
| 证据`evidence` | 62 | 应报发现和记录的召回率、发现与记录能否对应 |
| 制度`policy` | 45 | 制度版本、条款位置和适用关系是否正确 |
| 推理`reasoning` | 74 | 金额、日期、发票、聚合口径及事实到结论的推导是否成立 |
| 误报控制`false_positive` | 39 | 发现和记录精确率、无异常结论、例外和边界是否正确 |
| 格式`format` | 15 | 统一提交和Schema是否有效 |

判定方式包括100个确定性规则项、136个混合判断项和15个纯语义判断项。确定性项用于核对集合、数量、精确率、召回率和提交状态，规则结果无论0或1均直接定稿；混合项由Judge同时查看结构化诊断、最终答案和工具轨迹；纯语义项用于判断制度解释或综合报告是否真正覆盖题目要求。

每道题内部的非格式检查项等权，先形成该题语义得分：

\[
Q_i = \frac{\text{该题通过的非格式检查项数}}{\text{该题全部非格式检查项数}} \times 100
\]

再对15道题等权平均形成任务正确性`Q`。L3-009已从73项压缩为27项，所有题均不超过30项；格式项不进入`Q`，提交、Schema和任务完成情况由稳定性`S`统一评价。无异常题报告实质异常时单题封顶40分；批量题记录或发现精确率低于50%时封顶50分。

**示例：RA-CASE-002三记录拆分核查。** 题面只给出`R004209`一条线索，不提示异常类型、关联记录数量、日期窗口或合计金额。Ground Truth要求候选通过数据查询找到`R004209`、`R004210`和`R004211`：三条记录均属于员工`E0008`的`travel_lodging`费用，日期为2月26日、2月28日和3月2日，首尾相隔4天；三笔各3400元，合计10200元，达到现行部门总经理审批线10000元，构成1项`SPLIT`风险。三张发票互不相同，因此不能误判成重复报销。该题拆为17项，关键评分点如下：

| 检查项 | 维度 | 通过条件 | 判定方式 |
| --- | --- | --- | --- |
| 审计结论 | 结论 | 明确认定本案构成拆分报销风险，且无矛盾表述 | Judge |
| 异常类型 | 结论 | 全部已提交发现均属于`SPLIT` | 规则最终判定 |
| 异常数量 | 结论 | 已提交发现数量恰好为1 | 规则最终判定 |
| 三条记录完整性 | 证据 | `record_ids`分别包含三条目标记录 | 3个规则项分别判定 |
| 无额外记录 | 证据 | `record_ids`中不存在三条目标记录以外的记录 | 规则最终判定 |
| 聚合对象 | 推理 | 正确说明同一员工、同一费用类型 | Judge结合数据轨迹 |
| 日期窗口 | 推理 | 三个日期完整，首尾相隔4天且处于7天窗口内 | Judge结合数据轨迹 |
| 金额与合计 | 推理 | 三笔各3400元，合计10200元 | 2个Judge评分项 |
| 阈值比较 | 推理 | 正确与现行10000元审批线比较 | Judge结合制度轨迹 |
| 发票排除 | 误报控制 | 识别三张发票不同，不报告`DUP` | Judge结合发票轨迹 |
| 制度依据 | 制度 | 引用费用报销办法第十一条及授权管理办法附件二 | 2个Judge评分项 |
| 独立窗口调查 | 证据 | 工具轨迹证明按员工、类型和日期查全三条记录 | Judge只看工具或工作产物 |
| 统一提交 | 格式 | 提交回执为`accepted`且Schema校验通过 | 规则 |

例如，某候选正确判断拆分报销并给出10200元合计，但只返回两条记录、漏掉`R004211`，也没有证明执行过完整窗口查询，则对应的记录项和调查过程项分别记0；其余事实仍按各自检查项判定，不会因为异常方向正确而整题满分。

\[
Q_{RA-CASE-002} = \frac{14}{16} \times 100 = 87.5
\]

若统一提交成功，格式项记1，但不会提高该题的语义得分。这样既能识别候选已经找对异常方向，也能明确暴露其证据召回和事实说明不足，而不是将整题简单判为全对或全错。

## 六、评分与判卷机制

最终总分为100分：

\[
Total = 0.75Q + 0.15S + 0.10F
\]

其中：

- `Q`为任务正确性。先计算每题全部非格式Rubric项的命中率，再对15题取平均。`record_ids`、制度依据、推理和误报控制已经包含在逐题Rubric中，不再作为独立权重重复计分。
- `S`为稳定性。任务需在900秒内完成、进程正常退出、正式提交被接受且Schema有效；满足的任务数除以15。
- `F`为执行时效。以15题耗时中位数计分，180秒以内为100分，900秒为0分，中间线性下降。

证据质量仍单独展示`record_id`宏平均F1、制度评分点得分率、误报控制评分点得分率和陷阱题特异度，但只用于解释质量结构。Token因六个框架的缓存口径和采集覆盖率尚未完全一致，只作为诊断指标，不进入总分。

v7判卷由规则最终处理集合、数量、精确率、召回率、边界记录和Schema等确定性项；只有制度适用性、证据对应和业务推理进入DeepSeek Judge。Judge按最多4个Rubric项分批判断，固定抽样20%的语义项复判，分歧时第三次裁决。新版本使用独立的`gate5_grades_v7.jsonl`和`gate5_judge_v7/`，不得复用历史逐项成绩。

**Pi Agent。** B组Total为95.63，在18组中排名第二，全年审计达到100；C组Q下降10.9分，制度与版本、全年审计和综合报告同时回退。该框架应保留Agent按完整Skills编排的B组，不采用C组包装。

**OpenClaude。** B组已稳定提升，C组进一步把Q提高1.1分、Total提高0.8分，全年审计和综合报告分别达到84.4和96.5，15题零超时。B/C均值得进入业务复测，再以证据完整性决定包装方式。

**OpenCode。** A组Total为90.98；B组因范围扩大和报告收敛失败明显回退。C组相对B组恢复6.8分Q和7.4分Total，综合报告由67.2恢复至89.6，但仍未超过A组，适合作为脚本治理专项复测对象。

**Codex。** B组相对A组Q提高10.44分，主要来自全年审计由48.9升至97.3；C组Q回退3.7分，综合报告仍只有63.0，`L3-009`补跑后仍超时。当前保留B组技术对照，优先解决长报告收敛。

**Oh My Pi。** B组在陷阱和综合报告上较强，但全年审计降至36.1并出现3次超时；C组减少到1次超时，却未恢复业务质量。脚本治理改善完成率但没有解决全年核查能力，暂不进入首轮业务候选。

C组同时改变薄Skills、文件提交、可选子智能体和脚本校验，C-B只能解释为两套增强包装的整体差值，不是单因素因果效应。

### 7.4 B-A与C-A负收益归因

负收益需要区分两种情况：一是Q正确性实际下降；二是Q有所提高，但新增流程造成超时和耗时上升，最终Total下降。各框架相对A组的变化如下：

| 框架 | B-A Q | B-A Total | C-A Q | C-A Total | 主要判断 |
| --- | ---: | ---: | ---: | ---: | --- |
| Claude Code Best | +3.96 | +0.59 | +3.90 | +4.34 | 整体正收益，B组耗时上升抵消了部分正确性收益 |
| Codex | +10.44 | +6.69 | +6.70 | +6.02 | 整体正收益，但陷阱题和长报告仍有局部回退 |
| OpenClaude | +2.38 | +1.79 | +3.44 | +2.58 | 整体正收益，C组个别陷阱题语义判断回退 |
| OpenCode | -7.28 | -7.64 | -0.50 | -0.21 | B组出现范围与收敛问题；C组基本恢复到A组，但尚未形成确定净收益 |
| Oh My Pi | +2.17 | -3.55 | +0.58 | +0.44 | B组正确性略升，但被3次超时和耗时上升抵消 |
| Pi Agent | +5.80 | +3.61 | -5.14 | -3.85 | B组适配良好；C组在制度比较和全年审计中丢失语义要点 |

**第一类原因是增强流程超过框架的有效编排能力。** OpenCode B组在`L3-009`中加载完整Skills、维护Checkpoint并发起2次子智能体请求，900秒内未形成提交；A组在463秒完成。该题使B组综合报告得分从A组的89.6降至接近零，是OpenCode B-A负收益的最大来源。说明子智能体和治理步骤本身不会自动产生增益；当框架不能及时收敛子任务时，它们会占用完成主任务所需的时间预算。

**第二类原因是工具调用增加，但没有转化为更完整的业务结论。** Oh My Pi在`L3-004`的A组使用89次工具调用、745秒完成并取得100分；B组增加到122次工具调用，并执行数据核查子智能体，最终在901秒超时；C组进一步达到192次工具调用，其中`list_expenses`调用168次，仍未提交。其问题不是缺少审计步骤，而是重复分页代替了SQL或聚合查询。B组虽然整体Q提高2.17分，但稳定性和时效损失使Total下降3.55分。

**第三类原因是模型在增强流程中错误解释了题目范围，而现有增强机制未能纠偏。** OpenCode B组在全年超标准题`L3-003`中将任务缩窄为办公用品和通讯费，只返回2项发现，遗漏培训、招待和差旅等题目明确要求的费用类型，单题得分由A组69.2降至26.9。C组则走向另一端，把大量差旅记录按不完整口径判为异常，返回305条记录。这里的直接原因是模型范围判断错误，不是Skills或脚本主动修改了业务范围；B/C相对A的下降说明不同增强包装会影响模型的范围解释，并且当前流程缺少“题目要求与实际核查范围逐项对照”的强制校验。完整Skills或脚本路由只能规定检查步骤，若范围判断错误且未被校验拦截，流程越完整反而越可能稳定执行错误口径。

**第四类原因是结构完整不等于语义完整。** Pi Agent C组在`L3-008`中查询到了差旅和培训制度，但最终提交只保留一句未完成的制度说明，`citations`为空；Schema和统一提交仍然通过，单题得分由A组93.8降至37.5。C组脚本能够检查字段、集合和文件状态，却不能自动判断答案是否已经覆盖职级维度、三档培训标准、同档数值比较和禁止混用依据。这是Pi Agent C-A整体负收益的主要案例。

以上负收益不能简单解释为“Skills无效”或“脚本无效”。更准确的结论是：

1. Skills增加了业务方法，也增加了调用链和上下文负担，必须根据框架的编排能力控制触发数量。
2. 脚本适合承担状态、集合和提交校验，不适合代替制度适用、范围理解和报告完整性判断。
3. 对全年任务应优先提供SQL/Python聚合路径，并限制重复分页和无效子智能体调用。
4. 结果校验应在Schema之外增加题型级语义覆盖检查，例如制度比较题必须检查维度、数值、对照和排他依据是否全部出现。
5. 每类仅3题且每组只运行1次，OpenCode C-A Total的-0.21分应视为基本持平；下一轮需通过业务确认题和重复运行判断负收益是否稳定。

## 九、基于历史v5的暂定结论与复测建议

### 9.1 技术路线结论

推荐路线调整为“强Coding Agent + 可插拔Skills + 按框架选择的确定性脚本治理层”。成立的是能力分层机制和部分框架上的净收益，而不是同一增强包装对所有框架的普遍提升。当前可以确认：

1. 共享业务核心可被六个源码框架装配，制度、SQL、复核和提交能力不必重复开发。
2. Claude Code Best、Codex、OpenClaude和Pi Agent同时提升Q与Total，说明增强包在四个框架产生可量化净收益。
3. Oh My Pi只出现质量净收益，稳定性和时效回退；OpenCode当前增强组为负收益。
4. 将确定性步骤下沉后，Claude Code Best、OpenClaude和OpenCode形成Total净收益；Codex与Pi Agent回退，Oh My Pi仅改善完成率。脚本治理层应按框架和题型选择启用。
5. Skills应保留制度适用、数据边界、异常判断和报告语义等需要模型推理的内容；状态、校验和提交等可重复确定的动作可下沉脚本，但不能用脚本替代审计判断。

预设的“Q至少提升8分且五类中至少四类改善”只作为强提升诊断线，不是判断增强是否有意义的唯一标准。15题规模下，只要总分形成稳定正向变化并改善关键复杂题型，就已经是值得继续验证的信号。

### 9.2 候选顺序

**第一候选：Claude Code Best脚本增强组与原增强组。** C组Q保持95.59、Total达到96.68且零超时；B组Q为95.65。两种包装应在业务确认题上并行复测，判断C组的时效收益能否稳定保持且证据质量不回退。

**第二候选：Pi Agent B组。** 该配置在18组中排名第二，且全年审计能力最强；同框架C组明显回退，不建议切换脚本下沉包装。重点验证陷阱题回退是否稳定存在。

**第三候选：OpenClaude原增强组与脚本增强组。** 两组正确性、稳定性和时效均衡，C组Q和Total均小幅提升且零超时，适合按证据完整性进一步选择。

**专项候选：OpenCode脚本增强组。** C组修复了原增强组的大部分负优化，Q提升6.8分、Total提升7.4分，接近基线组，适合继续验证是否可稳定超过基线。

Codex和Oh My Pi暂不进入第一轮业务候选。Codex保留原增强组作为技术对照；Oh My Pi应继续修复全年任务的调用链与长任务收敛。

## 十、下一步执行项

1. 先完成4道替换Case的12组区分度Canary；每题需满足得分极差至少15分、标准差至少5分且满分组不超过9组。通过后再由业务人员逐题确认15题的题意、制度口径、Ground Truth和251项Rubric，形成签字或评审记录。

4. 各适配器统一上报上下文估算方法、压缩前后Token和保留字段校验结果，再将原生上下文压缩纳入比较。
5. 修复Pi Agent陷阱题回退、Claude Code Best路由拒绝、OpenClaude制度引用、Codex综合报告超时、OpenCode残余范围扩张和Oh My Pi全年超时后，再决定最终框架及其增强包装。
