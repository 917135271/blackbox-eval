# 证券费用审计智能体框架领域增强调研报告

**适用项目：** 证券费用审计、制度问答与业务数据核查智能体  
**评测对象：** Claude Code Best、Codex、OpenClaude、OpenCode、Oh My Pi  
**统一模型：** DeepSeek V4 Pro  
**正式实验：** 5个框架、10组配置、15个正式案例、150次任务运行  
**报告日期：** 2026年7月

## Executive Summary（执行摘要）

- **建议将 OpenClaude 增强组作为下一轮业务验证的第一候选，将 Claude Code Best 增强组作为重点对照候选。** OpenClaude 增强组在本轮取得最高综合分79.81分、任务正确性79.63分，15题全部完成提交且无超时；Claude Code Best增强组在单案核查和综合报告方面表现稳定，适合继续验证其插件化路线。

- **“强 Coding Agent + 可插拔 Skills”的技术路线成立，但现有增强包尚未达到可以直接定型的程度。** 五个框架的增强组都没有达到预设的“任务正确性提升至少8分且至少四类题型改善”标准。Skills已经证明能够改善单案聚焦、反向核查和综合报告，但全年批量审计、记录范围控制和运行稳定性仍需修复。

- **本轮框架差异主要体现在任务编排和上下文控制，而不是基础制度问答能力。** 五个框架在制度与版本判断、单案数据核查上普遍表现较好；真正拉开差距的是全年扫描、陷阱题、综合报告、超时和Token消耗。Codex增强组出现3题超时，平均非缓存输入加输出Token约131万，说明当前DeepSeek适配下存在明显的上下文累积问题。

- **子智能体不是本轮主要增益来源。** OpenClaude增强组的24次子智能体请求全部被路由规则拒绝，仍取得增强组第一；Oh My Pi增强组完成9次子智能体调用，但正确性反而低于基线。下一阶段应优先优化Skills、任务状态、专项范围和结果校验，再针对确有并行调查或独立复核需要的复杂题启用子智能体。

综合判断，本项目不应直接选择一个框架后大规模开发，也不应退回从零自研Agent Loop。更合适的路线是：

> 以 OpenClaude 或 Claude Code Best 作为通用执行底座，保留共享审计核心和统一治理层，先修复专项扫描、陷阱反查和上下文预算，再进行一次冻结后的确认性复测。

## 一、调研背景与结论修正

上一轮标准55题黑盒评测结果为：

| 候选 | 正确题数 | 正确率 |
| --- | ---: | ---: |
| trae-agent | 45/55 | 81.8% |
| OpenCode | 41/55 | 74.5% |
| qwen-code | 33/55 | 60.0% |
| Claude Code | 32/55 | 58.2% |
| Codex | 31/55 | 56.4% |

上一轮所有候选格式成功率均为100%，没有超时和Judge错误，说明题库、MCP、结果采集和判卷链路已经具备基础可用性。但该结果不能直接解释为Claude Code和Codex框架能力较弱，原因是当时采用统一DeepSeek后端、MCP-only工具模式、最多12步、受限文件和Shell能力，并要求智能体直接生成严格JSON。Claude侧使用第三方`claude-code-best@2.8.3`，Codex通过自建协议代理连接DeepSeek，二者都没有按完整Coding Agent方式运行。

因此，上一轮应定义为：

> 统一DeepSeek后端、受限工具和有限步骤条件下的兼容性基线。

上一轮轨迹真正暴露的问题主要是审计过程不完整，而不是JSON格式错误：

- 查询范围没有先规划，查到少量记录后过早下结论；
- 没有充分使用SQL或Python执行全量核查；
- 已查询到相关记录，但最终遗漏`record_ids`；
- 历史、现行和废止制度之间的适用关系判断不完整；
- 无异常题缺少反向验证，容易基于相似记录误报；
- 综合报告中的事实、制度、判断和影响没有完整对应。

本轮调研由此从“比较Agent Loop”调整为“比较完整可落地智能体方案”，并验证是否可以在强Coding Agent上通过Skills、MCP和治理层快速形成证券审计能力。

## 二、为什么采用 Coding Agent + Skills

证券费用审计任务同时需要制度检索、版本判断、SQL聚合、Python复算、文件管理、证据整理和结构化提交。Coding Agent天然具备多轮规划、Shell、文件操作和工具编排能力，适合作为通用执行底座。若从零开发Agent Loop，需要重新实现工具协议、工作区、失败恢复、上下文管理、非交互执行和轨迹留存，工作量较大，且很难在短期内达到成熟Coding Agent的执行能力。

另一方面，单纯依赖框架通用能力也不足以完成稳定审计。制度适用、异常口径、证据覆盖和无异常反查都属于业务方法，必须通过可检查、可修改的领域能力进行约束。因此，本轮采用：

> Coding Agent + 领域Skills + 单题任务状态 + 按需子智能体 + 统一Hooks审计 + 阶段化上下文压缩

该路线的必要性体现在三个方面：

1. **复用成熟执行能力。** 框架继续负责任务规划、Shell、Python、SQLite、文件工作区和工具调用，不重复建设通用执行引擎。
2. **业务能力可插拔。** 制度版本判断、批量审计、证据检查和报告生成分别沉淀为Skill。业务规则调整时优先修改对应Skill、参考资料或脚本，不需要重写框架主循环。
3. **框架可以替换。** 审计规则、SQL方法、路由条件和结果Schema放在共享业务核心中，五个框架只维护薄适配层，减少对单一框架的绑定。

本轮设计参考了插件和Skills的组织方式，并结合上一轮真实调用轨迹形成，不是根据正式题答案反向编写。开发题和正式题使用独立记录命名空间，正式任务、Rubric和Ground Truth在GATE3冻结前不向候选开放。

## 四、领域增强能力设计

共享审计核心包含七个Skills。每个Skill定义触发条件、执行步骤、工具范围、输出产物、停止条件和失败处理，避免将全部业务要求堆入一个长提示词。

| Skill | 主要职责 | 解决的上一轮问题 |
| --- | --- | --- |
| `audit-query-planner` | 明确对象、期间、规则、字段、证据和专项范围 | 查询范围不完整、过早下结论 |
| `policy-version-check` | 判断现行、历史、废止、替代和过渡制度 | 旧制度误用、版本关系不清 |
| `batch-expense-analysis` | 使用SQL/Python执行全量扫描、聚合和复算 | 只检查样例记录 |
| `evidence-coverage-check` | 核对异常项、`record_ids`和制度依据是否完整 | 查询到记录但最终漏报 |
| `false-positive-review` | 对无异常、边界、例外和陷阱题执行反向验证 | 无异常题误报 |
| `audit-report` | 组织事实、规则、判断、影响和建议 | 报告结构和证据对应不足 |
| `result-validator` | 校验Schema、记录范围、引用和正文一致性 | 中间分析与最终结果不一致 |

增强组先执行审计规划，再按需执行制度核查和数据核查；形成证据矩阵后进行误报复核和报告生成，最后通过结果校验及统一提交工具完成提交。简单事实题允许精简执行，全年审计和综合报告必须生成批量查询或可复核中间产物。

### 运行治理总体结构

增强组不是只向提示词中加入Skills，而是在框架外增加统一治理层。该治理层由四部分组成：

1. **统一Runner**负责创建单题隔离工作区、启动和终止候选进程、执行900秒超时控制、汇总外部工具日志、生成运行清单和最终产物清单。无论候选是否支持原生Hooks，Runner都能记录任务开始、任务结束、进程失败和超时。
2. **审计控制MCP**向主智能体提供子智能体授权、子智能体完成登记、上下文Checkpoint、结果预校验和正式提交工具。权限、路径、Token预算、提交次数等关键约束在工具端校验，不能仅靠模型自觉遵守。
3. **框架原生能力**负责实际执行Shell、Python、SQLite、文件操作和原生子智能体。各框架的调用方式可以不同，但必须通过统一控制MCP完成关键状态登记。
4. **文件化任务状态**保存审计计划、证据、判断、产物和上下文恢复信息，使运行状态不只存在于模型对话历史中。

四部分的职责边界是：Runner管进程和全局留痕，控制MCP管强制规则，原生框架管任务执行，Task Memory管状态持久化。

### Hooks的具体实现

Hooks采用“外层统一采集 + 内层业务事件”两层机制。

**第一层是统一Runner Hook。** 每题启动前，Runner调用`initialize_task_state`创建工作目录和初始状态，并写入`run_manifest.json`。任务结束时，Runner读取各框架产生的工具日志，将一次工具调用统一转换为`tool_started`和`tool_completed`或`tool_failed`事件，再写入最终状态、超时信息和产物哈希。即使候选中途崩溃，仍能保留进程级轨迹。

**第二层是控制MCP和框架原生Hook。** 主智能体在选择Skill、申请子智能体、写入Checkpoint、执行结果校验和提交答案时产生更细的业务事件。事件统一写入以下文件：

| 文件 | 记录内容 |
| --- | --- |
| `traces/events.jsonl` | 按序号保存全部统一事件 |
| `traces/tool_calls.jsonl` | 工具名称、参数字段、执行结果和错误 |
| `traces/subagents.jsonl` | 子智能体申请、授权、拒绝和完成事件 |
| `traces/context_events.jsonl` | Checkpoint和压缩事件 |
| `traces/artifact_manifest.json` | 产物路径、SHA-256、文件大小、责任角色和调用ID |
| `run_manifest.json` | 框架、实验组、模型、开始结束时间、返回码、超时和关键产物 |

统一事件至少包含任务ID、框架、实验组、角色、时间、递增序号、事件类型、来源、原因码、输入输出摘要、产物路径、Token、耗时和错误码。当前支持的关键事件包括：

- 任务类：`task_started`、`task_completed`、`task_timeout`；
- Skill及工具类：`skill_selected`、`tool_started`、`tool_completed`、`tool_failed`；
- 子智能体类：`subagent_requested`、`subagent_authorized`、`subagent_rejected`、`subagent_completed`；
- 上下文类：`context_checkpoint`、`context_compacted`；
- 校验提交类：`validation_started`、`validation_completed`、`submission_attempted`、`submission_accepted`、`submission_rejected`；
- 产物类：`artifact_written`。

Hooks执行时采用以下工程约束：

1. 每条事件在文件锁内取得单调递增序号，降低并行写入导致的乱序风险。
2. JSON状态文件采用临时文件加原子替换方式写入，避免进程中断留下半个JSON。
3. 所有产物路径必须解析后仍位于当前题目工作区内；路径越界或文件不存在时拒绝登记。
4. 日志摘要写入前执行密钥特征脱敏，检测到类似`sk-...`的内容时替换为`[REDACTED]`。
5. 子智能体产物写入清单时计算SHA-256和文件大小，可检查产物是否在登记后被修改。
6. 权限授权、子智能体闭环、结果校验和正式提交属于强制事件，失败时阻断后续流程；普通统计日志异常可记录为基础设施故障，但不能改变业务答案。

需要说明的是，事件Schema已经支持`skill_selected`，但不同框架对原生Skill生命周期的暴露程度不同。本轮正式运行中，任务、工具、子智能体、上下文和提交事件已经统一采集，Skill选择事件尚未在五个框架中做到同等粒度的自动捕获。后续应由各适配器在进入Skill时显式调用统一事件工具，才能准确计算Skill实际触发率。

### Task Memory的具体实现

本方案中的Memory不是跨任务长期记忆，也不使用向量数据库，而是当前单题工作区内的结构化文件。它解决的是模型在多轮工具调用后容易忘记范围、证据和未决事项的问题。

Memory分为三类：

| 类型 | 内容 | 正式评测规则 |
| --- | --- | --- |
| 领域记忆 | Skills、制度解释方法、SQL模板、Schema和工作流程 | GATE3后冻结，各题只读共享 |
| 任务工作记忆 | 当前题目的计划、证据、决定、产物和剩余预算 | 允许读写，仅在当前题有效 |
| 跨任务记忆 | 其他题答案、历史轨迹、失败经验和标准答案 | 明确禁止读取和共享 |

每题初始化时创建五个核心状态文件：

| 文件 | 作用 | 典型更新时机 |
| --- | --- | --- |
| `work/task_state.json` | 保存当前阶段、范围、预算、已完成步骤、待办事项、未决问题和提交状态 | 每个审计阶段完成后 |
| `work/evidence_index.json` | 按记录、制度和发现索引证据，避免重复查询和漏掉已找到证据 | 制度查询、数据查询或证据补充后 |
| `work/decision_log.jsonl` | 追加记录关键判断、责任角色、原因和未决事项 | 范围冻结、子智能体返回、发现保留或驳回时 |
| `work/artifact_index.json` | 保存SQL、结果集、制度分析、复核意见等产物的位置、哈希和所有者 | 新产物完成登记时 |
| `work/context_checkpoint.json` | 保存可恢复的最小上下文包 | 阶段切换、达到容量阈值或提交前 |

这些文件不是简单备份，而是后续步骤的输入。例如，独立复核员启动前必须存在`work/evidence_matrix.json`；结果提交前会检查是否仍有状态为`authorized`但未完成登记的子智能体；主智能体在压缩上下文后根据`context_checkpoint.json`恢复题目范围、制度、记录ID、未决事项和剩余预算。

单题工作区同时保存审计业务产物：

- `work/audit_plan.json`：任务拆解、范围、复杂度和Skill路由；
- `work/policy_applicability.json`：适用制度、排除制度和期间；
- `work/analysis/data_findings.json`：SQL/Python核查结果；
- `work/evidence_matrix.json`：每项发现的事实证据、制度证据和覆盖状态；
- `work/independent_review.json`：复核员的保留、驳回和补证意见；
- `work/report_draft.json`、`work/validation_report.json`和`work/final_result.json`：报告、校验和最终提交前结果。

这样设计后，对话历史不再是事实来源。模型可以遗忘已完成工具调用的原始文本，但不能丢失已写入状态文件的结论、证据位置和决策记录。

### 子智能体的具体实现

子智能体用于职责隔离、并行调查和独立复核，不以“上下文太长”作为单独触发理由。简单查一条制度或执行一条SQL时，由主智能体调用Skill完成，避免无效分派。

三个角色的触发条件、权限和产出如下：

| 子智能体 | 允许触发的典型原因 | Token上限 | 必须产出 |
| --- | --- | --- | --- |
| 制度研究员 | 多版本核对、历史制度、制度冲突、期间不清、多条款组合 | 4000 | 适用和排除制度、期间、条款、未决问题及制度分析文件 |
| 数据核查员 | 聚合、多表关联、多记录范围、日期金额边界、Python复算 | 5000 | SQL/Python脚本、完整结果集、统计口径、候选记录和异常记录 |
| 独立复核员 | 例外、阈值边界、跨期、制度数据冲突、证据缺口、综合审计、无异常复核 | 4000 | 对每项发现的保留、驳回或补证意见，以及遗漏和误报清单 |

路由先由`audit-query-planner`给出0至6的复杂度：

- 复杂度0至1：不允许调用专业子智能体，也不允许调用复核员；
- 复杂度2至3：最多调用一个制度研究员或数据核查员，可按条件调用复核员；
- 复杂度4至6：最多调用两个专业角色，可按条件调用复核员；
- 每个角色每题最多授权一次，子智能体不得再次创建子智能体。

一次完整调用必须经过以下步骤：

1. 主智能体先完成`work/audit_plan.json`，明确角色、复杂度、原因码、任务边界、输入产物和Token预算。
2. 主智能体调用`authorize_audit_subagent`。控制MCP检查角色是否存在、原因码是否允许、复杂度是否满足、角色调用次数、Token预算、上下文长度、是否包含禁用资产或密钥，以及输入路径是否位于当前工作区。
3. 授权通过后，控制MCP生成唯一`invocation_id`，创建`work/subagents/<role>/`目录，并返回角色工作目录、Token上限、允许输入和必须返回的字段。此时状态记为`authorized`。
4. 框架使用自身原生机制实际启动子智能体。主智能体可以并行处理不重叠的工作，但不得重复执行已委派的制度或数据调查。
5. 子智能体把完整过程保存在角色目录，包括SQL、Python、结果集、制度摘录或复核记录；返回主智能体的摘要只包含`decision`、`key_findings`、`record_ids`、`citations`、`unresolved_items`和`artifact_paths`。
6. 主智能体调用`complete_audit_subagent`登记完成。控制MCP检查摘要字段是否完整、是否存在额外字段、摘要是否超过6000字符、是否包含密钥、产物是否位于该角色目录、文件是否存在，以及Token是否超过授权预算。
7. 完成登记后，系统计算产物哈希，更新`artifact_index.json`和`decision_log.jsonl`，写入`subagent_completed`事件，并将状态改为`completed`。主智能体只把结构化摘要放入活跃上下文，需要核实细节时再按路径读取完整产物。

该两阶段协议的关键作用是区分“模型说自己调用过子智能体”和“系统确认子智能体已经完成并留下可审计产物”。如果存在已授权但未完成登记的调用，`submit_audit_result`会返回`SUBAGENT_COMPLETION_REQUIRED`并拒绝提交。这样可以防止主智能体在子任务未返回时直接给出结论。

独立复核员与普通信息调查角色不同。它在主智能体形成初步发现和`work/evidence_matrix.json`后启动，仅接收题目范围、初步发现、证据矩阵和必要产物路径，不需要接收主智能体的完整推理对话。这样可以降低结论锚定，但由于主智能体和复核员仍使用同一DeepSeek模型，其独立性属于角色和上下文隔离，不代表模型层面的完全独立。

### 上下文管理与压缩的具体实现

上下文管理采用“阶段Checkpoint + 容量阈值 + 文件回读”三部分，而不是把全部历史对话持续塞入模型。

阶段Checkpoint包括：

| 阶段 | 写入时机 | 主要目的 |
| --- | --- | --- |
| `task_started` | 工作区初始化后 | 固定题目、权限和预算 |
| `planning_completed` | 审计计划完成后 | 固定范围、规则、复杂度和路由 |
| `investigation_completed` | 制度和数据调查结束后 | 固定候选发现、记录和制度 |
| `review_ready` | 证据矩阵完成、复核前 | 为独立复核提供稳定输入 |
| `validation_ready` | 报告完成、提交校验前 | 固定最终证据和答案草稿 |
| `submission_ready` | 正式提交前或Runner收尾时 | 保存可恢复的最终状态 |

每个Checkpoint必须保留题目、限制条件、审计计划、适用制度、候选或最终`record_ids`、证据状态、未决问题、产物索引、剩余预算和提交状态。引用的产物路径必须真实存在且位于当前工作区。

容量阈值按以下方式执行：

1. 估算上下文使用达到60%时，写入一次完整Checkpoint。之后即使模型会话中断，也可以根据文件恢复。
2. 达到75%时，先将已完成的工具对话、长制度原文、SQL原始行和子智能体完整过程写入文件，再把活跃上下文压缩为Checkpoint中的结构化状态与必要摘要，并记录`context_compacted`事件。
3. 达到85%时停止宽泛检索和无边界探索，只允许执行定向补证、证据核对、结果校验和提交，防止在接近上下文上限时再次扩张任务范围。

压缩时可以从活跃上下文移除：

- 已完成工具调用的完整输入输出；
- 已保存到文件的SQL原始结果；
- 已引用的制度长原文；
- 子智能体完整调查过程；
- 已被后续结论替代的中间草稿。

压缩时不得移除：

- 原始题目和专项范围；
- 禁止事项和时间、Token预算；
- 适用制度及版本关系；
- 候选和最终记录ID；
- 每项发现的证据覆盖状态；
- 未解决问题；
- 产物路径和哈希；
- 当前阶段、剩余步骤和提交状态。

控制MCP会校验Checkpoint阶段、使用率范围、必需字段和产物路径，更新`task_state.json`及内部控制状态，并记录Checkpoint次数和压缩次数。Runner在增强组结束时还会检查是否存在可恢复的最终Checkpoint；若候选没有生成合格的`submission_ready`状态，Runner会依据最终答案、提交回执、证据矩阵、审计计划和运行预算重建一个收尾Checkpoint。

需要明确当前实现边界：统一治理层能够强制“Checkpoint内容完整、路径有效、事件留痕和最终可恢复”，但不能在所有第三方框架内部直接删除或重写模型已经持有的对话历史。60%、75%和85%由适配器根据框架可获得的Token统计或估算值触发；真正的原生上下文压缩仍依赖各框架自身能力。Runner收尾重建Checkpoint只证明任务状态可恢复，不等同于证明模型在运行中已经按75%阈值完成过原生压缩。因此，下一轮若要把上下文治理作为正式比较指标，应要求各适配器统一上报`estimation_method`、压缩前后Token和保留字段校验结果。

### 四类机制如何协同

单题执行时，四类机制按照以下关系协同：

1. Runner创建隔离工作区并写入任务开始事件。
2. 主智能体读取`audit-query-planner`，将计划和范围写入Task Memory。
3. 若计划满足复杂度和原因码，控制MCP授权子智能体；子智能体完成后以摘要和产物回传。
4. 主智能体将制度、数据和子智能体结果写入证据索引和证据矩阵，而不是长期保留全部原始对话。
5. 每个阶段写Checkpoint；达到容量阈值时从Task Memory重建精简上下文。
6. 结果校验和提交Hook检查证据、路径、子智能体闭环和提交次数，不满足条件时阻断。
7. Runner在任务完成或超时时写入最终事件、运行状态和全部产物哈希。

因此，Skills回答“应该怎样做审计”，子智能体回答“由谁独立完成复杂子任务”，Task Memory回答“当前任务事实保存在哪里”，Context Compression回答“哪些内容继续进入模型”，Hooks回答“全过程如何被记录和强制约束”。五者职责不同，不能互相替代。

## 六、评分与判卷口径

最终总分为：

$Total = 0.55Q + 0.20E + 0.15S + 0.10F$

- `Q`为15道题逐题Rubric得分的平均值，权重55%；
- `E`衡量`record_ids`、制度引用和无异常题判断，权重20%；
- `S`衡量是否在时间内完成、成功提交且无运行中断，权重15%；
- `F`衡量中位耗时和非缓存输入加输出Token，权重10%。

每题Rubric均在正式运行前冻结，按照结论、证据、制度、推理、误报控制和格式分别评分。判卷采用级联方式：

1. Schema、超时、提交状态和集合匹配由规则程序计算；
2. 规则判满分的客观项直接保留，不重复调用Judge；
3. 规则判错、缺失或不完整的项目交给DeepSeek Judge复核，以Judge结果为准；
4. 制度适用、推理质量和报告完整性等语义项直接由Judge评分；
5. 未提交或超时且没有有效结果的任务直接记0分。

该方式避免规则对等价异常ID、合理记录范围或自然语言表达产生机械误判，同时保留集合和运行状态的客观计算能力。150条结果全部完成判卷，不存在规则负判未交Judge的项目，Judge错误为0。

## 九、各题型表现与典型轨迹

### 9.1 制度与版本判断已经不是主要瓶颈

十组制度与版本题平均分均超过92分。各框架基本能够识别现行审批线、废止版本以及差旅住宿和培训住宿的制度差异。该类题主要依赖检索和制度关系表达，Coding Agent的通用文件读取与MCP能力已经足够。

这意味着下一阶段没有必要继续增加大量简单制度问答题，而应将制度能力重点放在多版本冲突、业务发生时点和跨制度引用链上。

### 9.2 单案核查最能体现Skills的正向价值

Claude Code Best在`L2-013`基线组得到60分。基线答案虽然准确识别了`R004223`培训费超标准，但同时扩展到全年6笔超标准记录，偏离了题目要求的单案范围。增强组通过审计规划和证据检查，只返回`R004223`，准确说明3700元超过3500元标准且无专项审批，得分提升到100分。

该案例证明，`audit-query-planner`和`evidence-coverage-check`能够改善“发现了正确异常但返回范围过大”的问题。单案核查不是查询越多越好，关键是锁定题目对象并保持证据集合精确。

### 9.3 全年批量审计是五个框架的共同短板

全年审计平均分显著低于制度和单案题。主要错误包括：

- 将“超标准专项扫描”扩展为重复报销、拆分、超期和预算等全规则扫描；
- 返回部门超预算后的全部记录，而不是Rubric要求的临界异常记录；
- 只检查部分月份或样例记录，却将结论表述为全年完整扫描；
- 中间查询结果过大，导致上下文和提交时间失控。

OpenClaude增强组在`L3-003`中将超标准专项错误扩展为9项异常，其中6项是重复发票，15条记录只命中3条目标记录，最终仅得15分。该失败说明当前`batch-expense-analysis`能够执行批量查询，但没有充分约束“只执行本题规则”和“最终记录集合必须与专项口径一致”。

因此，全年审计问题不是模型不会写SQL，而是缺少查询范围锁定、聚合口径和结果收敛机制。

### 9.4 陷阱题需要识别题目目标，而不是泛化扫描

OpenClaude增强组在`TRAP-003`中准确核查`R004236`和`R004237`相隔8天，超过7日拆分聚合窗口，返回“无异常”，得分100分。Oh My Pi基线组在`TRAP-002`中正确判断不同发票且合计低于审批线，但将全部8条陷阱记录作为证据提交，因范围过大得到92.5分。

相反，Oh My Pi增强组在同一道`TRAP-002`中错误核查了`R000002`和`R004201`，把题目目标映射到另一组重复发票数据，只得到15分。该问题说明增强流程在某些情况下增加了宽泛检索和错误锚定，Task Memory保存了错误目标后，后续证据检查和报告流程只能强化错误结论。

下一阶段需要在审计计划生成后加入“目标记录锁定”检查，禁止在没有明确理由时替换题目指定记录。

### 9.5 综合报告受益于结构化Skills，但数量核对仍不足

OpenCode基线组在`L3-009`中遗漏超标准规则，并把审批缺失当作第五类异常，报告声称涉及42条记录，得分62分。增强组能够覆盖五类规则并形成更完整报告，得分提升到78分，但仍把异常数和背景记录范围扩大，没有准确说明30个异常和5个陷阱位。

OpenClaude增强组和Claude Code Best增强组的综合报告题型均达到92.3分，说明`audit-report`能够有效改善报告结构、制度引用和规则覆盖。但`audit-report`不能替代数据结果核对，最终异常数量和记录范围仍应由独立的结果校验逻辑确定。

## 十、框架差异分析

### OpenClaude：当前最均衡的完整方案

OpenClaude增强组Total 79.81分、Q 79.63分，15题全部提交、0超时，中位耗时199秒。其优势是制度、单案和综合报告能力较均衡，增强组在陷阱题和综合报告上有明确提升，且运行稳定。

需要注意的是，OpenClaude增强组24次子智能体请求全部被路由拒绝，说明其成绩主要来自主智能体、Skills、任务状态和结果校验，而不是子智能体。OpenClaude相对基线Q只提升4.53分，尚不能证明现有插件包已充分发挥作用。

### Claude Code Best：增强收益明确，但执行时间偏长

Claude Code Best增强组Q提升4.47分，单案核查达到100分，综合报告达到92.3分，说明其Skills加载和业务工作流执行有效。其主要问题是中位耗时521秒、2题超时，增强后全年审计反而下降8.33分。

该框架适合作为重点对照候选，下一阶段需精简增强提示、减少无效子智能体请求，并将批量结果写入文件后只向主上下文返回摘要。

### Codex：基础能力较强，但当前DeepSeek适配效率不足

Codex基线组制度与版本题98.3分，单案核查88.3分，说明框架基础推理和工具调用能力并不弱。增强组单案核查提升到100分，但出现3题超时，稳定性降至80分，平均可比Token约131万，效率分仅33.52。

当前结果主要反映Codex通过Responses-to-Chat代理连接DeepSeek时的上下文和协议适配问题，不能外推为Codex使用原生模型时的能力结论。在修复Token累积、自动压缩和子智能体完成登记之前，不建议将Codex增强组作为当前项目主框架。

### OpenCode：运行轻量，综合报告提升明显

OpenCode基线组中位耗时138秒、平均Token约3.5万，效率得分100，是本轮最轻量的执行方案。增强组将综合报告提高30.33分，单案核查达到100分，但出现2题超时，全年审计和陷阱题下降，导致Total低于基线。

OpenCode适合作为轻量备选和工程对照，但需要修复增强流程对简单题和陷阱题的过度调用。

### Oh My Pi：基线强，现有增强包与其原生机制冲突

Oh My Pi基线组Total 79.36分，是表现最好的基线组，制度、单案和陷阱题能力较强。增强组完成9次子智能体调用，但Q下降3.53分、Total下降6.89分，说明统一增强流程与其原生任务管理、Memory或工具编排存在重复和冲突。

Oh My Pi不应直接套用与其他框架完全相同的增强包装。若继续评估，应保留共享业务规则，但重新设计适配层，减少重复规划、重复复核和上下文扩张。

## 十一、子智能体与上下文治理结论

本轮子智能体调用情况如下：

| 增强组 | 请求 | 授权 | 完成 | 拒绝 |
| --- | ---: | ---: | ---: | ---: |
| Claude Code Best | 21 | 2 | 2 | 19 |
| Codex | 8 | 7 | 3 | 1 |
| OpenClaude | 24 | 0 | 0 | 24 |
| OpenCode | 7 | 7 | 5 | 0 |
| Oh My Pi | 9 | 9 | 9 | 0 |

调用次数与最终得分没有正相关关系。OpenClaude没有实际执行子智能体仍取得第一；Oh My Pi执行最多且完整完成，但增强效果为负；Codex存在授权后未完成登记，并伴随超时。

因此，子智能体的必要性应限定为：

- 制度与数据调查可以真实并行；
- 需要不同工具权限或独立产物；
- 需要独立复核员尝试推翻主结论；
- 单项调查可以在独立Token预算内完成。

如果只是读取一条制度、执行一条SQL或减少上下文长度，应优先使用主智能体、Skill、任务状态和Checkpoint，不调用子智能体。下一阶段应将“简单题禁止调用”“只有可并行任务才授权”“授权后必须预留完成时间”写入硬路由规则。

## 十二、技术路线判断

本轮结果支持继续采用“Coding Agent + Skills”，原因不是所有增强组都取得高提升，而是以下事实已经得到验证：

1. 同一份共享业务核心能够成功包装到五个不同框架；
2. 单案核查和综合报告出现了可定位、可复现的增强收益；
3. Skills、Hooks、任务状态和统一提交可以独立留痕并进行失败归因；
4. 框架更换时无需重写制度、SQL方法、证据标准和结果Schema；
5. 增强失败主要可以定位到具体Skill、路由和上下文策略，而不是只能修改一个不可解释的长提示词。

同时，本轮也否定了两个过于乐观的假设：

- 安装更多Skills和子智能体不会自动提高正确率；
- 一套完全相同的增强流程不能不加调整地适用于所有Coding Agent。

因此，后续路线应从“统一增强包”调整为：

> 共享业务核心保持一致，框架适配层根据原生任务管理、上下文和子智能体机制进行差异化装配。

## 十三、选型建议

### 第一候选：OpenClaude增强组

推荐作为下一轮真实业务流程验证底座，理由是：

- 本轮Q和Total均为第一；
- 15题全部提交，0超时；
- 单案核查100分，综合报告92.3分；
- 增强流程没有依赖大量子智能体，运行路径相对可控；
- 基线组本身表现稳定，降低了对插件效果波动的依赖。

### 重点对照：Claude Code Best增强组

建议保留为重点对照，理由是：

- 单案核查和综合报告提升明确；
- Skills和受控工作流较容易表达和维护；
- 与“强Coding Agent + 可插拔能力”的目标路线一致。

其进入主选的前提是将增强组超时从2题降至不超过1题，并解决全年专项扫描范围扩张问题。

### 暂不作为主选：Codex增强组

Codex不应因本轮排名被排除，但当前DeepSeek适配下的Token和超时问题会直接影响项目可用性。建议先进行协议和上下文专项优化，再用相同15题复测，不应在当前状态下直接作为主框架。

### 备选观察：Oh My Pi基线与OpenCode

Oh My Pi基线可作为“少增强、强框架能力”路线的对照；OpenCode可作为轻量、低耗时方案的对照。二者都不建议直接采用当前统一增强包装。

## 十四、下一阶段修复与复测计划

下一阶段不扩大题量，优先修复以下问题：

1. **专项范围锁定。** `audit-query-planner`必须输出唯一规则、目标记录或扫描口径；`batch-expense-analysis`不得自动扩展到其他异常类型。
2. **超预算记录定义。** 明确异常记录是首次越过预算的临界记录、全部超预算后记录，还是部门级异常；Schema和Rubric使用同一口径。
3. **陷阱题目标校验。** 题目指定记录必须进入`task_state.json`并在提交前核对，禁止无理由替换为相似发票或其他陷阱记录。
4. **批量结果外置。** SQL原始行和全年结果保存到文件，主上下文只保留统计摘要、目标ID和未决问题。
5. **子智能体收缩。** 简单题禁止调用；复杂题最多调用真正需要的角色；任务剩余时间不足时不得新建子智能体。
6. **Codex上下文专项治理。** 检查Responses代理的累计消息、工具结果回传和自动压缩，目标是将增强组平均Token和超时降到可比范围。
7. **差异化适配。** Oh My Pi和OpenCode保留共享规则，但不强制完全相同的规划、Memory和子智能体流程。

修复完成后，继续使用同一15道冻结正式案例进行确认性复测。建议成功标准为：

- 每组超时不超过1题；
- OpenClaude和Claude Code Best增强组Q相对基线提升至少8分；
- 全年审计题型平均分至少提高15分；
- 陷阱题平均分至少达到70分；
- `record_ids`完全正确题数明显提升；
- 不增加新的格式错误、Judge错误和轨迹泄漏。

## 十六、限制与待确认事项

1. 本轮统一使用DeepSeek V4 Pro，结论反映“框架 + DeepSeek适配 + 领域能力”的完整方案，不代表各框架使用原生模型时的绝对能力排名。
2. 正式集为15题，每类3题，适合定位典型能力和失败原因，但不足以证明所有证券审计场景的统计稳定性。
3. 每个正式案例每组只运行一次，结果中仍可能包含模型单次采样波动。
4. Judge同样使用DeepSeek。规则判满分的客观项直接保留，规则负判和语义项由Judge最终裁决，可以减少机械误判，但仍可能存在同模型偏差。
5. 当前OpenClaude增强组虽然排名第一，但增强收益未达到预设门槛。建议将其定义为下一轮第一候选，而不是已经完成最终生产选型。

## 最终结论

本轮调研证明，直接比较受限Agent Loop不能准确反映Coding Agent在证券审计任务中的价值；将成熟Coding Agent与可插拔业务Skills、统一MCP和治理层结合，是一条能够降低开发工作量、保留框架替换能力并支持审计追溯的可行路线。

当前推荐顺序为：

1. **OpenClaude增强组：下一轮第一候选；**
2. **Claude Code Best增强组：重点对照候选；**
3. **Oh My Pi基线组：强框架能力参考；**
4. **OpenCode：轻量效率参考；**
5. **Codex：完成DeepSeek协议和上下文优化后再评估。**

在完成专项扫描范围、陷阱题目标锁定、超预算口径和上下文预算修复前，不建议直接进入大规模业务开发。下一阶段应以同一15题确认性复测为准，验证增强组是否能够稳定、明确地超过自身基线。
